# 충돌 해소를 정적 규칙이 아닌 판례(Precedent) 학습 루프로 모델링

상태: accepted (2026-06-20)

담당이 깨끗하게 정해지지 않는 Conflict(후보가 여럿인 **Overlap**, 후보가 없는 **Gap**)는 업무의 그레이 영역이라 한 번에 규칙으로 닫을 수 없다. 그래서 충돌을 사람이 해소하게 하고(Overlap은 후보 Owner 합의 → 실패 시 Manager, Gap은 Manager가 지정 또는 신규 카드 생성), 그 합의를 append-only **Precedent**로 기록해 라우터가 이후 유사 케이스에 참조하게 한다. 라우터의 행동 = 선언된 규칙 + 누적된 판례.

## Consequences

- append-only 감사 로그가 곧 판례 저장소다 — 별도 시스템이 아니다.
- Precedent는 그대로 라우팅 회귀 테스트 케이스가 된다(학습 루프 = TDD 스위트).
- 불변식: **어떤 질문도 미아로 남지 않는다.** 0건 매칭 → 루트 Manager로 Escalation.
- 반복되는 Gap은 "아무도 안 맡는 영역" backlog로 드러나 신규 Agent Card 필요 신호가 된다.
- 합의는 카드를 파괴적으로 수정하지 않는다 — 카드 경계 수정은 판례의 선택적 후속 결과일 뿐이다.
- Manager는 팀/도메인별 계층(`escalates_to`) + 루트 기본 Manager로 표현하며, 공통 상위를 찾는 트리 등반(LCA)은 후순위로 미룬다.

## 보강 (2026-06-20, T4.1)

위 "append-only 감사 로그가 곧 판례 저장소다 — 별도 시스템이 아니다"의 *의도*는 **판례에 새 영속 인프라를 들이지 않는다**(같은 append-only 메커니즘으로 충분, 외부 DB·검색엔진 불필요)는 것이지, 판례와 감사 기록을 한 자료구조에 섞으라는 강제가 아니다. T4.1은 이를 지키면서 별도 `PrecedentStore` 포트(`conflict.py`)를 둔다. 근거:

- **관심사·독자가 다르다.** 감사 로그(`AuditLog`)는 *운영자向* — 한 질문 처리의 전체 절차를 시간순으로 흘려보내는 기록이고, 같은 질문이 또 와도 한 줄이 더 쌓일 뿐 조회 키가 없다. 판례(`PrecedentStore`)는 *라우터向* — `lookup(intent)`로 합의된 결론을 끌어와 다음 라우팅에 적용하는, intent로 색인된 조회 가능한 사실이다.
- **결합하면 라우터가 감사 로그 스캔에 묶인다.** 감사 한 줄에서 합의 결론을 재구성하려면 라우터가 감사 포맷·전체 이력을 알아야 해 "전이 ≠ 기록" 경계가 무너진다. 별도 포트는 라우터가 결론(`Precedent`)만 보게 한다.
- **정신은 그대로 보존.** `InMemoryPrecedentStore`는 새 인프라가 아니라 append-only 리스트일 뿐이고, 영속이 필요하면 감사 로그와 동일하게 `JsonlPrecedentStore`로 같은 메커니즘을 재사용한다. "판례 = 라우팅 회귀 테스트 케이스"도 그대로다 — `record`된 `Resolution`을 시드해 라우팅을 검증한다.

따라서 ADR 0002의 결론(충돌→사람 합의→append-only 판례 학습 루프)은 불변이고, 그 판례의 *접근 포트만* 감사 로그와 분리해 명시한다.
