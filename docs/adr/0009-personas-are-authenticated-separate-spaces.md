# 페르소나별 면은 인증으로 분리된 별개 공간 — walking skeleton의 owner 선택은 임시 시연 장치

상태: accepted (2026-06-20)

PRD §4의 네 페르소나(실 사용자·개인 Owner·Manager·운영자)가 보는 화면(surface)은 서로 다른 *별개의 공간*이다. 실 사용자(채팅)와 운영 면(Owner 처리함·빌더, Manager 큐, 운영 모니터링)은 같은 사람이 아니며 권한·책임 경계가 다르다. 따라서 **최종 제품은 인증으로 페르소나/본인 식별을 강제한다** — Owner는 로그인해 *자기 소유 카드*의 처리함만 보고, 실 사용자 채팅과는 분리된 공간으로 진입한다.

walking skeleton(v0)은 인증이 후순위(PRD §6)라, 처리함의 owner를 드롭다운으로 고르는 **시연 장치로 "로그인을 가장"**한다. 이는 임시이며 완료 기준이 아니다. 백엔드(`/inbox/{owner_id}`·`/cases/{case_id}/concur` 등 owner별 엔드포인트)와 화면(`inbox.html` ≠ `index.html`)은 이미 페르소나별로 분리돼 있어, 인증만 끼우면 진짜 분리가 완성된다. "모두 같은 백엔드 공유"(TRD §8)는 유지된다 — 분리되는 것은 *접근 주체*이지 백엔드가 아니다.

## Consequences

- **최종 완료 기준**: (1) 실 사용자 면과 운영 면이 인증으로 분리된 공간, (2) Owner는 자기 소유 카드의 처리함만 접근, (3) `inbox.html`의 owner 가장 드롭다운 제거(세션의 인증된 `owner_id`로 대체).
- v0 시연 장치(owner `<select>`)는 `inbox.html`에 "데모용 로그인 가장"으로 명시한다 — 진짜 동작으로 오해하지 않게.
- 인증/분리 작업은 `tasks-v0.md` Phase 6(T6.5)에 둔다(SSO·정식 인증, PRD §6 후순위 항목과 연결).
- 페르소나 혼입(한 화면에서 여러 면을 임의 전환)은 권한·책임 경계를 흐리므로 최종 제품에선 금지 — 시연 단계라도 "이건 가장"임을 분명히 한다.
