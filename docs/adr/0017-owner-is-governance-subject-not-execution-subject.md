# Owner는 에이전트의 *관리 주체(거버넌스)*이지 *실행 주체*가 아니다 — 중앙이 owner 통제 지식으로 실행하고, 분산은 사설 데이터 한정 옵션

상태: accepted (2026-06-21) · **ADR 0010의 재정의(번복 아님, 분리)** · ADR 0011·0012(분산 전송·백업 워커)를 *기본 경로*에서 *선택 옵션*으로 재포지셔닝 · **→ 결정 2 ADR 0027로 재정의(2026-06-26·Phase 9)**: 아래 결정 2("답 실행 = 중앙 `claude -p`")의 *대화 답변 경로*가 owner OAuth 멀티-LLM 인프로세스 실행으로 재정의된다(분류기/배치 `claude -p`는 잔존). 0017이 owner측 실행에 든 반대(가용성·신선도·UX)는 백업 워커(0012)·Manager escalation(0014)·HITL 토글(0025)·인프로세스 속도가 흡수(ADR 0027). ADR 0011·0012 분산 WS 전송은 0027에서 *기본 대화 경로*로 다시 1급으로 재부상. **"중앙 키/토큰 0"은 0027에서도 보존·강화**(owner OAuth라 중앙 모델 토큰 0). 아래 "관리 ≠ 실행" 통찰·일곱 거버넌스 능력 표는 여전히 유효(owner 주권은 정의·범위·승인·검토·소유를 쥠에서 나옴 — 0024 세션·0025 HITL이 거버넌스를 더 강화).

## 맥락 — 끊어진 고리

ADR 0010은 "답변 주체 = 각 Owner의 Claude Code"로 못박으며 *최종 그림*을 **각 Owner PC의 분산 Claude Code 실행**(T6.3)으로 그렸다. 그 추론의 고리는:

> "owner가 자기 지식을 유지·관리한다 → **그러니까** owner 머신이 답을 실행해야 한다 → 그러니까 분산"

가운데 화살표가 헛점이다. **"owner가 관리한다" ≠ "owner가 실행한다".** 제품 비전을 다시 보면 핵심은 *실행 위치*가 아니라 **owner가 자기 에이전트의 관리 주체**라는 것 — owner가 정의·지식·범위·승인·검토·소유를 통제한다는 *거버넌스*다. 이건 중앙이 실행해도 100% 유지된다.

분산을 *강제*하는 유일한 요구는 "**중앙이 owner 지식을 읽는 것조차 안 된다**"(하드 데이터 격리)뿐인데, 지식 모델이 *공유 가능한 마크다운(OKF 번들)*인 현 설계에선 이는 기술적 필연이 아니라 *정책 선택*이다. 게다가 분산은 가용성(owner PC는 잠든다)·신선도·UX를 악화시키고, 가용성 폴백인 백업 워커(ADR 0012)는 결국 *owner 위임 인스턴스를 중앙 호스팅*해 "분산으로 가용성을 얻는다"는 명제를 스스로 배신한다(가용하려면 중앙화).

## 결정

### 1. Owner = 관리 주체(거버넌스), 실행 주체 아님

"owner가 자기 에이전트를 관리한다"를 능력으로 분해하면 **여덟 중 일곱이 거버넌스**이고 전부 중앙 실행과 양립한다:

| owner가 관리하는 것 | owner 주권 확보 | 분산 필요? |
|---|---|:---:|
| **정의** — domains·can/cannot·approval·페르소나 | owner가 카드를 *소유*·편집(아래 2) | ✕ |
| **지식 유지·최신화** | owner가 OKF를 *소유*, 중앙은 *최신을 읽기만* | ✕ |
| **범위 통제**(거부·승인) | 중앙이 owner *선언 규칙*을 집행 | ✕ |
| **답 검토·정정** | owner가 자기 이름으로 나간 답을 검토(`BackupReview`) | ✕ |
| **충돌 시 담당 결정** | owner 1인칭 합의 → 판례 | ✕ |
| **신원·책임 귀속** | `answered_by`=owner·audit·git author | ✕ |
| **라이프사이클**(활성/이관) | 중앙 레지스트리를 owner가 통제(Transfer) | ✕ |
| 지식이 중앙에 *절대* 안 읽혀야 | owner 환경 격리 실행 | **⭕**(유일) |

→ **owner의 힘은 런타임을 호스팅해서가 아니라 *정의·규칙·검토·소유를 쥐고 있어서* 나온다.** 중앙은 owner의 **대행자(deputy)**이지 대체자가 아니다 — 중앙은 카드 정의를 못 바꾸고, 선언된 범위를 못 넘고, 승인을 못 우회하고, 소유권을 못 가져간다.

### 2. 답 실행 = 중앙 Claude Code가 owner 통제 지식의 *최신*을 읽어 답한다

> ⚠️ **ADR 0027 (2026-06-26·Phase 9)이 결정 2를 재정의** — *대화 답변 경로*의 답 실행을 "중앙 `claude -p`가 owner OKF 최신을 읽기"에서 **owner OAuth 멀티-LLM 인프로세스 스트리밍**(owner 워커가 owner OAuth 구독 토큰으로 공급자 API[claude·codex·gemini 등]를 직접 부르고 토큰 스트리밍·프로세스 스폰 회피·속도)으로 되돌린다. *왜 다시 owner측 실행인가*: 사용자의 명시적 제품 비전(Hermes/opencode — 멀티-LLM OAuth 전용 채팅)·속도(인프로세스 vs `claude -p` 스폰)·멀티 공급자가 본질상 owner 자격증명을 요구. 결정 2가 든 반대(가용성·신선도·UX)는 백업 워커(0012)·Manager escalation(0014)·HITL 토글(0025)·인프로세스 속도가 흡수(ADR 0027). **"중앙 API 키 LLM RAG 회피·중앙 키 0"은 보존·강화**(자격증명이 owner OAuth라 중앙 모델 토큰 0). *분류기·배치 경로의 `claude -p`는 잔존*(대화 답변 경로만 교체). 아래 본문은 *그때의 추론 기록*으로 보존.

- 답은 **중앙**에서 `claude -p` 헤드리스로, cwd = 그 담당의 **OKF 디렉터리(특정 커밋 스냅샷)**를 두고 만든다(`ClaudeCodeRuntime`, T6.1·T6.7 그대로). 중앙 API 키 LLM RAG는 **여전히 안 쓴다**(ADR 0010의 유효한 근거 보존 — 로컬 `claude` 인증, 키·비용·모델 운영 회피).
- "자기 지식으로 답"은 *owner가 소유·유지하는 지식이 답을 grounding*함으로써 성립한다(실행 위치가 아니라 **지식 출처**가 owner). 답은 owner에 귀속되고 owner가 검토한다.
- **"중앙 무지식" 재해석**: 중앙은 지식의 *소유자·진실 원천이 아니다*(owner가 소유·유지, 편집 채널은 git/PR). 중앙은 답변 시 *최신을 읽을* 뿐이며 지식을 모아 RAG 인덱스로 들고 있지 않는다. "중앙이 읽는 것조차 금지"가 진짜 요구인 팀은 아래 4의 사설 데이터 옵션으로 간다.

### 3. 살아있는 지식 = git 저장(버전·범위·리뷰) + 빌더 UI 편집 + 신선도·변경 전파(1급)

- **저장**: OKF 번들을 git에 둔다(owner별 repo 또는 모노repo+CODEOWNERS). git은 버전·이력·owner 범위 쓰기·PR 리뷰·변경 웹훅을 *공짜로* 주고, 이미 편집 채널이다(CONTEXT Maintainer).
- **편집 UX는 git과 분리**: 비개발자 owner의 잦은 업데이트를 위해 **Agent 빌더 UI가 owner 대신 커밋**한다(owner는 git을 몰라도 됨). git의 견고함 + 폼의 마찰 0. (또는 팀이 이미 쓰는 Notion/Confluence 동기화 — 채택 마찰↓.)
- **실행 디렉터리**: agent별 *디렉터리*면 충분하다 — 하위폴더(`okf/{agent_id}/`, 현재) 또는 owner별 repo. **워크트리 불필요**(여러 브랜치 동시 체크아웃 문제는 우리에게 없다). 답변 일관성은 *커밋 스냅샷*(읽기전용)으로 — "이 답은 이 커밋 기준" 감사까지.
- **신선도·변경 전파를 1급으로**(이게 "살아있음"의 심장): 지식 변경의 ① *미래 답* 전파는 쉽다(최신 읽기). ② *과거 판례·답* 전파가 어렵고 핵심이다 — 정책이 바뀌면 그 정책에 기댄 **Precedent·답을 자동 재검토/무효화 플래그**. `last_reviewed_at`를 신뢰 신호로 노출, stale 지식은 owner에 갱신 nudge.

### 4. 분산 전송(ADR 0011·0012, T6.3/T6.6) = *사설 데이터 커넥터* 한정 옵션으로 강등

- 답이 owner의 **사설·실시간 데이터/도구**(자기 DB·메일·사내 API·자격증명)에 의존해 *중앙이 그 데이터를 가질 수 없을* 때에 한해, 그 담당이 **MCP 도구/커넥터**로 *데이터 접근만* owner 쪽에 노출하고 중앙이 호출한다(옵션 B). "답 생성 전체를 노트북에 보내기"가 아니다.
- 이미 구현된 분산 전송은 **버리지 않는다** — (a) owner-실행 경로가 *가능함*을 입증했고(사설 데이터 케이스의 토대), (b) 그 디스패처·작업 큐·재연결·멱등 인프라는 아래 6의 **비동기 처리·푸시 통지 인프라로 재포지셔닝**된다.

### 5. Owner 종속(binding) = card.owner(중앙 선언) + 실 신원(SSO)

owner를 특정 에이전트에 묶는 사슬은 *선언(중앙)·강제(편집 경계)·증명(신원)*:
- **선언**: `AgentCard.owner`를 레지스트리가 검증(Authority 중앙·불변식 — 에이전트가 자기 owner를 자처 못 함).
- **강제**: 편집은 빌더 스코프(세션 신원 ≠ card.owner → 403) 또는 git CODEOWNERS.
- **증명(미싱피스)**: 지금 무비밀번호 로그인이라 누구나 "cs_lead"로 들어온다 → **SSO/OIDC**(회사 신원 = `user_id`)가 종속의 *이빨*이다. T6.5에서 "후속"으로 미뤄둔 SSO가 사실 owner 주권의 *완성 조건*이었다.

### 6. 실시간 충돌 처리(1급)

"충돌이 실시간으로 owner에게 가고, 정리되면 자동 처리되는 중앙장치"는 이미 만든 라우팅→`ConflictCase`→1인칭 합의→`Precedent`→자동 라우팅 루프 + 두 추가(둘 다 중앙):
- **실시간 푸시 통지** — 지금 처리함은 *조회(pull)*뿐. 충돌·검토·escalation을 Slack/메일/MCP 알림으로 *push*(분산에서 만든 비동기 인프라 재사용).
- **변경 시 재평가** — 지식·담당이 바뀌면 관련 판례를 다시 열거나 무효화(위 3②와 연결).

## 근거

- **owner 주권의 정확한 정의** — "관리 주체"는 통제권이지 호스팅이 아니다. 일곱 거버넌스 능력은 중앙 실행과 양립하고 대부분 이미 구현됐다.
- **비전 정합** — 살아있는 지식·실시간 충돌 자동해소는 본질상 *중앙*(사용자 표현 "중앙장치") 문제다. 중앙이 빠르고(24/7)·최신을 읽고·판례로 학습할 때 자연스럽다.
- **분산의 비용** — 가용성·신선도·UX 악화 + 백업의 자기모순(가용하려면 중앙 호스팅). 이득(데이터 격리)은 *공유 마크다운*엔 해당 없음.
- **ADR 0010의 유효 근거 보존** — 중앙 API 키 LLM RAG 회피(`claude -p` 로컬 인증)는 그대로. 바뀐 건 "실행 *위치*가 owner여야 한다"는 부분뿐.

## Consequences

- **ADR 0010 재정의**: "답변 주체 = owner Claude Code"를 "답의 *지식 출처·책임*이 owner이고 *실행*은 중앙 `claude -p`가 owner 통제 지식으로"로 분리. ADR 0010의 "최종(T6.3) 분산"은 *기본 경로에서 제외*(옵션 4로 이동).
- **ADR 0011·0012 재포지셔닝**: 분산 전송·백업 워커는 *기본 아키텍처*가 아니라 *사설 데이터 커넥터 옵션*. 구현물은 보존·인프라 재사용(6).
- **로드맵 전환(tasks Phase 7)** — 우선순위: ① **SSO 신원 binding**(owner 종속 실재화) ② **git 저장 + 빌더 UI 편집 파이프라인**(살아있는 지식) ③ **지식 신선도·변경 전파**(판례 무효화 포함) ④ **실시간 충돌 푸시 통지** ⑤ 분산→옵션 B(커넥터)로 문서화·강등.
- **불변식 영향 없음** — 미아 없음·Authority 중앙·전이≠기록·등록 무결성·노출 불변식은 그대로. owner 주권은 오히려 *강화*된다(거버넌스 + SSO).
- **갱신 대상**: CONTEXT(Agent Runtime·Owner·Knowledge Bundle 절 — 관리/실행 분리), PRD §3 원칙·§5·§6, TRD §2·§4·§5, tasks(Phase 7 신설·T6.3/T6.6 재포지셔닝 주석). ADR 0010·0011·0012는 본 ADR로의 포인터를 단다.
- **미해결로 남김**: "중앙이 지식을 읽는 것조차 금지"가 우리 팀의 진짜 요구인지(→옵션 B 비중) **→ ADR 0020에서 진입 판단 게이트(세 질문)로 명문화**·변경 전파의 정확한 알고리즘(어느 판례를 무효화하나) **→ ADR 0019**·SSO 공급자 선택은 후속 결정**(→ T7.1)**.
