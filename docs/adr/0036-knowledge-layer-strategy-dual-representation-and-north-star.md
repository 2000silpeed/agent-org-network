# ADR 0036 — 지식 계층 전략: TrustGraph/EKOS 미채택 · 이중 표현 원칙 · 북극성(타입화 저작 진화)

- 상태: 채택(Accepted)
- 날짜: 2026-07-06 (사용자 승격 승인)
- 근거 본체: [`docs/trustgraph-eval-2026-07-06.md`](../trustgraph-eval-2026-07-06.md)(3자 비교·이중표현 원칙 §9-1·디리스크 스파이크 2라운드 §9-2·9-3·종합 판정 §9-4) · [`docs/scale-eval-2026-07-02.md`](../scale-eval-2026-07-02.md)(S11 커버리지 가설 기각)
- 계보: ADR 0028(published 인덱스·중앙 목차만·본문 0)·ADR 0029/0030(OKF 자동 저작·`LlmAuthor`·저작 토폴로지)·ADR 0032(의미 dedup·`Embedder`)·ADR 0033(중앙 답변·`Knowledge Store`)의 **후속(supersede 아님 — 지식 계층 *방향*을 박음)**. 단기 액션은 tasks `assemble_context`(T9.1b) 백로그.

## 맥락 — 지식이 대규모로 커질 때 어디로 가나

v0가 완성되고 운영화 단계로 오면서 두 외부 그래프 자산을 평가했다: [TrustGraph](https://github.com/trustgraph-ai/trustgraph)(외부 OSS·GraphRAG 인프라)와 EKOS(`~/ai-projects/Enterprise-knowledge-Operating-System` — 사용자 소유 온톨로지 그래프). 사용자 맥락은 (1) "RAG 없이는 완전한 라우팅이 안 된다"는 경험치, (2) 좋은 레포라 탐색, (3) 지식 대규모 확장 가정, (4) EKOS도 온톨로지 그래프 사상이라 함께 비교였다.

이 평가는 **되돌리기 어려운 전략 방향**을 낳는다 — 지식 계층을 외부 플랫폼에 얹을지, 자체 진화시킬지, 그리고 그 진화의 *설계 원칙*을 무엇으로 삼을지. 근거 문서에서 실측·스파이크로 확정한 결론을 정식 ADR로 박는다. **이건 지식 계층 방향이지 라우팅/실행 계층 재정의가 아니다** — 4대 불변식(미아 없음·무효 카드 거부·Authority 중앙·전이≠기록)과 ADR 0033의 "중앙 토큰 0 폐기·중앙 답변"은 이 ADR이 건드리지 않는다(§불변식).

근거 문서가 결정의 본체다. 이 ADR은 그 서술을 되풀이하지 않고 **결정·근거·결과**만 박는다.

## 결정

### 결정 1 — TrustGraph 미채택 (최후 수단으로만 잔존)

**지금 채택하지 않는다.** 실측·구조 근거:

- **라우팅 병목은 지식 깊이가 아니다** — scale-eval S11이 "unowned의 태반은 지식 커버리지 문제" 가설을 실패 36건 전수 판정으로 기각했다(커버리지 갭 1건뿐·보강해도 수치 무변화). 라우팅 지렛대는 매처·body 접지 어세서·하이브리드 리랭크였지 지식 확장이 아니었다.
- **답변 통증은 컨텍스트 초과가 아니다** — 에이전트당 본문 ~6.4k 토큰, 200k 윈도까지 ~30배 여유(실측). retrieval의 근거가 아니다. 진짜 통증은 **"라우팅 × 단일 에이전트 접지"의 구조적 경계**(근거가 두 도메인에 갈리는 관계형 질문이 절반만 답됨).
- **경량 사이드카 불가** — `trustgraph-base`가 Pulsar·Kafka를 하드 의존, GraphRAG 프로세서가 Flow 런타임(Pulsar)에 강결합. 단일 머신 docker-compose는 되나 "경량"이 아니라 무거운 풀스택 1식(JVM·Cassandra·Pulsar 상시). 우리 원칙(단일 머신·SQLite·경량)과 충돌.
- **2인 코어 버스팩터** + patch 러시(breaking 위험) + 상용 오픈코어 의도(핵심 폐쇄 가능성).

**대신**: 필요해지면 플랫폼 도입이나 별도 레포가 아니라 **GraphRAG 알고리즘(5단계 — 개념분해→벡터 seed→hop-and-filter 2-hop→출처추적→합성)만 차용**해 우리 `KnowledgeRetriever` 포트 뒤 자체 구현한다. 라이선스 Apache 2.0이라 차용·참조 안전(고지 보존만).

### 결정 2 — EKOS 미통합 · "빌릴 자매"

**as-is 통합하지 않는다.** EKOS의 강점(결정론 그래프·환각 0)은 **타입화 소스**(ABAP·DDIC·GUI 세션)를 파서가 읽는 데서 나온다. 우리 지식은 **OKF 마크다운 산문**(환불정책·근로 조문)이라:

1. EKOS 파서(ABAP 토크나이저·RFC·GUI)가 적용 안 됨 — 산문에서 관계를 뽑으려면 결국 **LLM 추출이 필요한데, 그건 EKOS가 의도적으로 안 하는 것**(환각 회피가 EKOS의 존재 이유).
2. EKOS 관계 9종은 **기술적 인과**(READS/WRITES/TRIGGERS)다. 우리 관계 통증은 **법적/정책적**(이 조항은 저 법 위반으로 무효·환불정책↔청약철회권)이라 EKOS Core 어휘가 아니다.

표면은 닮았지만 기질·관계·구축법이 다르다. **그러나 검증된 패턴 3종은 흡수한다**(별도 유지·한 가족):

- **evidence + confidence 필수** → `Answer Record`/출처·신뢰(책임)에 매핑·강화.
- **bitemporal(valid_from/to)** → `Correction Event`·staleness(ADR 0019 reeval)의 원리적 상위판.
- **닫힌 상위 온톨로지 + 열린 subtype** → `KnowledgeIndex` 관계 확장(라우팅 계층 가지치기)의 설계 패턴.

### 결정 3 — 이중 표현(Dual Representation) 설계 원칙 (핵심·되돌리기 어려움)

지식 계층 진화의 **불변 설계 원칙**을 박는다: **사람에겐 직관, 시스템엔 정확 — 산문과 타입화 그래프를 evidence로 결속한 하나의 지식.**

- **산문 = 진실 원천(Source of Truth)** — 사람 축·owner가 저작/검토(ADR 0013·0018·0029).
- **타입화 그래프 = 파생(Derived)** — 시스템 축·독립 저작 금지. **단방향 파생**(산문 → 그래프)이라 그래프는 산문보다 많을 수 없다(= **안전 경계**·환각 상한). 그래프가 산문에 없는 사실을 담으면 그건 파생이 아니라 창작이다.
- **Evidence 포인터 = 결속(Binding)** — 각 타입화 사실이 원문 span을 가리킨다. **삼역**: ① 감사(추적 가능성) ② 사람↔시스템 결속(같은 지식의 두 얼굴을 잇는 다리) ③ 환각 방어(원문에 없는 사실은 span을 못 가리켜 걸러짐).

**환각 문제와 해독제(정직한 화해):** owner는 구조화 사실을 직접 못 쓴다(현실) → **자동 변환(LLM 추출) 필수**. 나이브 추출은 TrustGraph의 환각이 된다. 해독제 = **owner 검토(HITL) + evidence 접지** — 우리가 이미 가진 것이다(**Phase 11 `LlmAuthor` 패턴의 출력 진화**·ADR 0029 초안→검토→확정 정신). 세 시스템은 이 축에서 화해한다:

| 시스템 | 추출 | 검토 | 근거 | 산문 |
|---|---|---|---|---|
| TrustGraph | LLM 추출 | ✗ | ✗(환각) | ✓ |
| EKOS | 파서(결정론) | — | ✓ | ✗(타입 소스 전제) |
| **AON (우리)** | **LLM 추출** | **owner HITL** | **evidence span** | **✓(산문 가능·감사 가능)** |

우리만 "산문에서 출발하되 환각 없이 타입화한다" — 추출 + 검토 + 근거의 조합으로.

### 결정 4 — 북극성: 기술 실현성 검증됨 · 빌드는 연기

지식 규모·관계형 답변 통증이 커지면 원리적 경로는 TrustGraph도 EKOS-import도 아니라 **OKF 저작을 "타입화 evidence-backed 사실"로 진화**(ADR 0029/0030 저작면 확장)시키는 것이다. 그러면 산문 대신 저작 단계에서 결정론 그래프가 나오고, 라우팅·답변이 공짜로 그래프를 얻으며, 그때 비로소 EKOS 엔진이 진짜 재사용 가능해진다.

**디리스크 스파이크 2라운드(2026-07-06·실 claude·스크래치패드·프로덕션 무변경)로 최난 기술 내기가 검증됐다:**

- **근거 접지 100%**(span 46/46 원문 exact·환각 0·닫힌 어휘 위반 0) — "evidence 결속으로 결정론 대체" 원리 작동.
- **장문 무열화**(라운드2 — 장문 span 접지 14/14·재현율 누락 0·최대 리스크였던 장문이 버팀).
- **재현 안정**(3문서×3회·분절 ±1뿐·내용 누락/추가 0).
- **크럭스 실증** — 그래프 엣지(COMPLEMENTS)가 절반 답을 완전한 답으로 전환. "환불불가면 청약철회 안 되나?" 질문에 D1(단일 접지)=절반(청약철회권만)·D2(이중 접지·엣지가 약관규제법 문서 견인)=완전(청약철회권 + 조항 무효 + 이중 요건). D2의 약관규제법 축은 D1 컨텍스트에 **물리적 부재** → 그래프 엣지가 견인해야만 나온다. 실 답변으로 실증.

**"될까?"는 풀렸고, 남은 미지는 제품/스케일**(스파이크가 아니라 실 빌드가 답한다):

1. **실 owner 산문의 지저분함**(오탈자·불완전·모순·비격식 — 스파이크 코퍼스는 깨끗한 5문서).
2. **문서 쌍 후보 생성**(스파이크는 손으로 짝지음 — 프로덕션은 "어느 문서를 짝지을지"가 별도 문제).
3. **owner 검토 UX·부담**(문서당 수 사실 × 지식베이스 규모의 지속 검토).
4. **비용/지연**(문서당 LLM 호출 + 편집마다 재추출).

**착수 트리거**: 관계형 통증이 `assemble_context`를 넘어섬 / 그래프 지식이 제품 중심이 됨. 트리거 시 **첫 설계 질문 = ② 문서 쌍 후보 생성**(contested 후보집합 / 임베딩 이웃이 후보이나 미설계). 스파이크 트랙은 여기서 종료(더 스파이크하지 말 것 — 남은 미지는 실 빌드+실 사용자라야 답함).

### 결정 5 — 단기 불변: `assemble_context`가 첫 정답 · 에스컬레이션 사다리를 박는다

측정된 통증(경계 넘는 관계형)의 첫 정답은 외부가 아니라 **우리 자신의 미구현 `assemble_context`(T9.1b·다중 에이전트 접지)**다. 현재 `answer()`는 라우팅된 단일 에이전트 본문만 주입(`provider_runtime.py`)해 관계형 질문이 구조적으로 절반만 답된다. 인접 에이전트 문서도 함께 접지해 통증을 정조준한다.

**COMPLEMENTS 엣지가 이 단기 액션을 구동하는 다리다** — 추출된 COMPLEMENTS 엣지가 곧 "어느 인접 에이전트를 함께 접지할지"를 정한다(북극성 추출과 단기 액션이 맞물림·순차 아님).

**에스컬레이션 사다리를 결정으로 박는다**(각 단계는 앞이 부족함이 실측될 때만):

1. **`assemble_context`(지금·우리 것)** — 다중 에이전트 접지. 측정된 통증 정조준.
2. **`KnowledgeRetriever` 포트 + 청크 retrieval** — 단일 에이전트 지식이 비대해질 때(30배 뒤·fastembed 재사용·인프라 0).
3. **자체 경량 GraphRAG** — 나이브 다중 접지로 관계 추론이 안 될 때. TrustGraph 5단계 알고리즘 차용·우리 포트 뒤 자체 구현(~2~4주).
4. **TrustGraph 사이드카(HTTP)** — 최후. 자체 구축이 그들 풀스택 운영보다 비쌀 때만(우리 규모엔 비현실적).

**공통 선결**: 어느 단계든 `KnowledgeRetriever`(또는 접지 입력) 포트 추상화를 먼저 세우면 1↔2↔3↔4 교체가 자유롭다. 단 **첫 실제 코드 액션은 포트가 아니라 `assemble_context`**(지금 아픈 곳).

## 근거

- **실측이 방향을 정했다** — S11(커버리지 기각)·30배 컨텍스트 여유·D1/D2 다운스트림 실증. 가설이 아니라 측정으로 "지식 깊이가 아니라 경계 넘는 접지가 통증"임을 확정했다.
- **포트 뒤 자체 구현이 플랫폼보다 우세** — GraphRAG 5단계가 단순하고 우리 fastembed·스택 일관성이 있어, 무거운 풀스택 도입·2인 버스팩터 추종보다 자체 어댑터가 싸다.
- **이중 표현이 세 시스템을 화해시킨다** — TrustGraph(근거X)·EKOS(산문X) 각각의 결함을 우리는 추출+검토+근거로 메운다. 이건 새 기계가 아니라 `LlmAuthor`(ADR 0029)·`Answer Record` evidence·reeval의 조각 재조합이다.
- **북극성은 검증됐으나 제품 미지가 남아 연기** — 기술 내기는 스파이크로 닫혔고, 남은 건 실 빌드라야 답하므로 "준비된 옵션"으로 두고 트리거 대기가 정직하다.

## 계보 (기존 ADR과의 관계 — 재정의/계승)

**전부 계승이지 supersede 없음.** 이 ADR은 지식 계층 *방향*을 박을 뿐 기존 결정을 뒤집지 않는다.

- **ADR 0028(중앙 목차·본문 0) — 계승·강화.** 라우팅 인덱스는 여전히 목차만. 타입화 그래프도 라우팅 축에선 목차(개념·엣지)이지 본문이 아니다. EKOS "닫힌 상위+열린 subtype"이 `KnowledgeIndex` 관계 확장(§7)의 설계 패턴.
- **ADR 0029/0030(OKF 자동 저작·`LlmAuthor`·저작 토폴로지) — 북극성의 확장 지점.** 북극성(결정 4)은 이 저작면을 "타입화 evidence-backed 사실"로 진화시키는 것이다. `LlmAuthor` 패턴의 출력이 산문 초안에서 타입화 사실+span으로 진화. 비소유·중앙 토큰 0(저작 전체 owner측)은 그대로.
- **ADR 0032(의미 dedup·`Embedder`) — 문서 쌍 후보 생성의 재료.** 북극성 첫 설계 질문(② 문서 쌍 후보)의 "임베딩 이웃"이 곧 `Embedder`/`classify_dedup_candidates`의 pairwise cosine 자산이다(owner측·중앙 0).
- **ADR 0033(중앙 답변·`Knowledge Store`) — 무변경 계승.** 이 전략은 지식 *계층*이지 실행 계층이 아니다. 중앙 답변·중앙 토큰 0 폐기·워커=지식 공급자는 그대로. `assemble_context`(결정 5)는 중앙 런타임이 `Knowledge Store`를 소비할 때 *여러* agent_id 본문을 접지하는 확장이다(단일→다중).
- **ADR 0019(신선도·reeval) — bitemporal 흡수 대상.** EKOS valid_from/to가 `Correction Event`·staleness의 원리적 상위판(결정 2).

## 4대 불변식 자체점검 (충돌 없음 확인)

- **미아 없음 — 무관(보존).** 지식 계층 진화는 *접지 품질*이지 *라우팅 종착*이 아니다. `assemble_context`·GraphRAG·타입화 그래프 어느 것도 0매칭→Unowned/root escalation을 바꾸지 않는다.
- **유효하지 않은 카드는 등록되지 않는다 — 무관(보존).** admission 무변경. 타입화 사실도 evidence span 없이는 안 들어가는 게 "환각 방어 경계"(결정 3)로 오히려 admission 정신을 확장.
- **Authority 중앙 — 무관(보존).** 저작 자동화는 후보 생성이지 권한 생성이 아니다(ADR 0029 S6·`concept.domain ∈ card.domains` 검증). 타입화 그래프도 under-claim 자기보고이지 권한 선언이 아니다.
- **전이 ≠ 기록 — 보존.** evidence 포인터는 감사 축(기록), 산문→그래프 파생은 저작 축(전이). bitemporal 흡수도 이 분리(유효시간 vs 기록시간)를 원리화할 뿐 섞지 않는다.
- **"중앙 토큰 0 폐기"(ADR 0033) — 무충돌.** 이 전략은 지식 계층 방향이라 실행 계층(중앙 답변·조직 키)을 재정의하지 않는다. 북극성 저작 진화도 저작 전체가 owner측이라 저작 토큰은 여전히 owner(비소유 보존·강화).
