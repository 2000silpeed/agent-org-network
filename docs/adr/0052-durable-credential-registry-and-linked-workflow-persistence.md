# ADR 0052 — Durable Credential Registry와 linked workflow persistence

- 상태: 제안(Proposed)
- 날짜: 2026-07-15
- 계보: ADR 0043~0048의 Request-first 수명주기, ADR 0049~0051의 production authority·운영 MCP 경계를 P17.9의 durable transaction·outbox·lease·복구 계약으로 잇는다.
- 적용 범위: P17.9 persistence contract, first vertical slice인 Durable Credential Registry, operational approval·audit·outbox intent UoW, 이후 linked workflow migration의 순서.
- 제외 범위: production FastAPI/실 OIDC, 물리적 exactly-once, 실제 backup/restore·DR(P17.13), 전 테이블 일괄 재작성.

## 맥락

현재 `TokenStore`와 `SqliteTokenStore`는 legacy admission token이다. 조직, credential generation, conditional revoke, central worker binding, durable operation approval evidence가 없으므로 이를 운영 MCP credential로 승격하면 권한·철회·재연결 fence를 과장한다. P17.8 S5의 worker epoch fence도 같은 프로세스의 stale-session 방어일 뿐 재시작과 다중 인스턴스에서 공유되지 않는다.

## 결정

P17.9는 범용 저장소 교체가 아니라 linked workflow를 수직 슬라이스로 durable화한다. 첫 슬라이스는 새 `DurableCredentialRegistry`다. legacy `AdmissionToken`을 migrate authority로 쓰지 않는다.

`DurableCredential`은 `credential_id`, `org_id`, `owner_subject_id`, `role`, immutable `generation`, lifecycle status, expected revision과 issued/revoked/expiry metadata를 가진다. raw secret은 해시만 저장하고 발급 응답에서 한 번만 안전 전달한다. read/list/audit/outbox/MCP error에는 raw secret을 넣지 않는다.

issue/revoke는 서버 principal, current resource, canonical command digest와 operation approval evidence를 대조한 뒤 같은 DB transaction에서 credential state, command receipt, secret-free procedural audit intent, outbox intent를 함께 쓴다. 재시도는 같은 semantic idempotency key와 digest일 때만 immutable result receipt로 수렴하며, 다른 payload는 write 0이다. revoke는 current record·generation·revision을 직접 재읽고 CAS한다. credential MCP는 이 registry/UoW capability가 조립된 경우에만 등록되며, 그렇지 않으면 legacy fallback 없이 unavailable이다.

P17.9 central operational mutation receipt/audit/outbox intent도 current source scope proof의 `org_id`, source kind·instance reference·scope revision·snapshot digest와 resource revision을 exact 결박해야 한다. legacy source write-through, provenance 없는 행의 backfill, receipt 추정 repair는 금지한다. source proof가 없거나 drift하면 approval·ResourceRef grant가 있어도 mutation은 write 0이다.

credential issue의 안전 전달은 `stage(raw_secret, idempotency_key) → opaque delivery_ref → registry commit → release(delivery_ref)` 순서다. DB receipt와 outbox에는 reference만 기록한다. release 실패 재시도는 같은 persisted reference만 다시 release하며 raw secret을 재stage·재전달하지 않는다. stage 실패는 local write 0이고 commit 실패 뒤 staged reference abort는 best-effort다. external delivery의 durable confirmation과 physical exactly-once는 이 단계의 보장이 아니다.

정식 production/multi-instance target은 PostgreSQL이다. SQLite는 단일 인스턴스 결정론·conformance·migration 테스트 adapter로 남는다. DB time, conditional update와 lease claim은 PostgreSQL의 locking/skip-locked 동등 계약으로 증명한다. 실제 production server는 이 결정만으로 열리지 않는다.

correlation·유일 제약의 첫 조각으로 Approval-first SQLite schema capability를 도입했다. 이 migration은 검증된 Question Completion 부모 component 위에 `approval_items`와 `(request_id, approval_round)` 유일 제약, open item 하나를 강제하는 partial unique index, parent Request 및 supersession FK를 명시적으로 설치한다. manifest/catalog는 canonical digest로 검증하고 runtime open은 DDL 보정 없이 validate-only다. row-level reconciliation은 parent 조직, indexed mirror, supersession lineage를 읽기 전용으로 대조해 손상된 snapshot을 fail-closed한다.

Approval-first v1은 실행 source가 아니다. 기존 Approval generation의 identity는 `(request_id, attempt, approval_round)`이고 같은 Request의 새 attempt는 round 1부터 다시 시작한다. 따라서 durable operation은 v1을 바꾸지 않고 별 `durable_approval_assignments_v2` component에서 그 세 튜플을 identity로 선언한다. v1에서 v2로의 자동 이관·권한 승격·receipt replay는 없다. v2 terminal UoW는 current assignment, Question Request, 중앙 authorization evidence, Completion 결과, immutable command receipt, secret-free audit/outbox intent를 하나의 DB transaction으로 결박한다. raw question·candidate·edited body는 receipt/audit/outbox에 복제하지 않는다.

첫 실행 슬라이스는 durable approve·approve-with-edit·reject decision이다. command는 서버 principal과 action, item·generation·Request revision에 결박되고, receipt는 같은 semantic digest와 principal만 immutable 결과로 복원한다. 다른 command, 조직, 세대, revision, 권한은 write 0 conflict 또는 unavailable이다. 결정은 같은 SQLite write transaction 안에서 current assignment·Request를 다시 읽고, 중앙 grant를 write 직전에 재인가한 뒤 resolved assignment, Completion 결과 또는 Declined Request, receipt, 본문 없는 audit/outbox intent를 함께 쓴다. shared Completion connection은 typed ownership context와 writer-thread token으로 보호한다. 외부 callback은 public read/write로 미완료 상태를 볼 수 없고, internal planner만 exact context reader를 통해 같은 transaction의 assignment를 읽는다. 이 보장은 SQLite 단일 인스턴스·한 process 범위이며 delivery 소비·lease·외부 exactly-once는 포함하지 않는다.

다음 실행 슬라이스인 durable lifecycle은 기존 v2 assignment component와 decision component의 DDL을 바꾸지 않고 별 `durable_approval_lifecycle_v1` component로 설치한다. lifecycle receipt는 predecessor assignment마다 하나이고 command digest도 유일하다. receipt에는 `manual_reassignment | expiry_reassignment | expiry_unavailable` 중 하나의 sealed evidence snapshot과 `reassigned | unavailable` 결과 snapshot만 둔다. 질문·초안·수정 본문은 receipt·evidence·audit·outbox에 복제하지 않는다. 이미 commit된 receipt는 같은 principal·action·digest의 재시도에서만, 현재 중앙 권한을 다시 확인한 뒤 immutable 결과를 복원한다. 다른 payload, 조직, predecessor generation, Request revision은 write 0 conflict다.

S3.2a는 이 component의 schema capability까지만 구현했다. migration은 parent v2 capability를 먼저 대조하고 owned DDL·manifest를 하나의 fault-atomic transaction으로 설치한다. open과 reconciliation은 manifest/catalog·FK·companion row·parent lineage를 validate-only로 대조한다. evidence/result JSON은 duplicate key·비정규 표현·비유한 수를 거부하고, exact field 계약의 opaque identifier·lower SHA-256·allowlisted action/kind/failure/outbox enum·canonical timezone-aware ISO timestamp·SQLite 범위 정수만 받는다. 따라서 외부 hash를 다시 계산한 일반 문장도 저장된 snapshot을 유효하게 만들 수 없다. 이 완료는 lifecycle command, receipt replay, row repair, scheduler/lease 또는 delivery를 활성화하지 않는다.

manual lifecycle evidence는 central grant digest와 별도로 `ApprovalReassignmentAuthorization.authority_version`의 SHA-256을 가진다. replay/reconciliation은 이를 predecessor supersession과 successor requirement·due·created time·Request successor state에 exact 대조한다. 필드가 없는 이전 manual receipt는 호환 처리·backfill·추정 replay 없이 fail-closed한다. expiry는 transaction-scoped DB time을 한 번만 읽고, policy digest를 실제 predecessor/successor 또는 unavailable/Failed Request closure와 재구성해 대조한다. 이 보장은 SQLite 단일 인스턴스 UoW까지이며 scheduler, lease, delivery, multi-instance recovery를 포함하지 않는다.

one-shot reconciliation은 durable scheduler가 아니다. 현재 principal 조직의 due/open assignment ID를 read-only DB-time snapshot으로 한 번 고정한 뒤 각 ID를 fresh expiry UoW로 위임하며, 그 실행에서 새 successor를 다시 scan하지 않는다. schema/catalog/FK 검증은 공통 DB capability로 유지하지만 row reconciliation은 principal 조직에 scoped되어 다른 조직의 손상 행이 현재 조직 command의 가용성을 막지 못한다. cursor·cache·lease·claim·다중 인스턴스 보장은 S3.5 이후에만 연다.

S4는 legacy InMemory Conflict/Manager/dispatcher 및 full-secret claim·control handle을 durable authority로 이관하지 않는다. 새 linked aggregate component를 schema→direct Conflict→Conflict-to-Manager→source별 Manager→WorkTicket 순서로 연다. 각각 current ResourceRef 중앙 재인가, Request revision CAS, immutable receipt, 본문·secret 없는 intent를 같은 transaction에 결박해야 하며, legacy 이관·backfill·receipt 추정 repair는 없다. WorkTicket의 lease/claim/send/timeout scheduler와 delivery consumer는 S5 이전에 열지 않는다.

S4.1은 이 순서의 schema capability만 `durable_linked_aggregates_v1`로 구현한다. Completion parent capability를 먼저 확인한 뒤 ConflictCase·ManagerItem·WorkTicket compact snapshot, command receipt, audit/outbox intent의 six owned table과 canonical manifest를 하나의 fault-atomic migration으로 설치한다. 모든 persisted reference는 field-specific `kind:<lowercase SHA-256>`이어야 하며, receipt target reference는 declared target kind와 exact 대응한다. digest·allowlisted enum·SQLite 범위 nonnegative integer·실제 calendar와 offset을 parse/round-trip한 timestamp 이외의 scalar는 거부한다. 따라서 correct hash를 동반한 일반 문장, raw secret 또는 control-shaped reference도 runtime open과 조직 scoped reconciliation에서 repair 없이 fail-closed한다. schema/catalog/FK 검증은 DB 공통 capability로 전역 수행하되 row scalar/lineage 검증은 principal 조직 범위이며, 이 slice는 legacy promotion·backfill·Conflict/Manager/WorkTicket operation activation·lease·scheduler·delivery를 열지 않는다.

S4.2 direct Conflict는 S4.1 hash-only snapshot을 즉시 command authority로 쓰지 않는다. 후보 set digest와 command digest만으로는 Owner별 vote, round, 선택 Card 및 권한 근거를 재구성할 수 없기 때문이다. 먼저 별 `durable_direct_conflict_uow_v1` component가 append-only Owner vote·`conflict.concur` receipt·audit/outbox intent·canonical result projection을 fault-atomic하게 설치한다. `durable_linked_aggregates_v1`의 manifest/DDl은 변경하지 않는다. raw 후보·Owner/Card ID, rationale, 질문, grant/control token/secret은 transaction 동안 현재 Registry 검증과 ordered candidate-set digest 계산에만 쓰며 저장하지 않는다. 저장 값은 typed digest reference, hash, allowlisted enum, 범위 정수, 실제 calendar/offset timestamp로 제한한다.

그 뒤 concurrence UoW는 정확히 `conflict.concur` 중앙 action과 actor가 Owner인 current `conflict_case` ResourceRef를 시작과 write 직전에 다시 인가한다. Case open/current round와 `AwaitingConflict` Request revision, 현재 Registry의 후보 Card·Owner·domain·선택 target을 둘 다 재검증한다. 새 Owner vote는 immutable receipt와 함께 한 transaction으로 기록하며, 모든 현재 후보 Owner가 같은 target을 표결한 마지막 write만 Case resolve·Request `ReadyToDispatch(attempt=1)`·terminal receipt/audit/outbox를 같이 commit한다. 같은 receipt replay도 현재 권한과 Card/Owner를 재검증한다. divergent vote·Registry drift는 이 단계에서 Manager로 전이하지 않고 S4.3의 별 evidence 경계로 남긴다. legacy InMemory claim/control handle·`RequestRouteAuthority` 외부 write, WorkTicket enqueue/wake/delivery, lease/scheduler, vote 기반 추정 repair/recovery는 사용하지 않는다.

S4.2a는 위 UoW의 evidence schema만 완료했다. `durable_direct_conflict_uow_v1`은 Owner vote 하나와 `conflict.concur` receipt 하나, receipt별 audit/outbox intent와 result projection 하나를 FK·UNIQUE·row mirror로 정확히 결박한다. candidate set digest는 parent durable Conflict snapshot과 같아야 하고, receipt/projection의 selected Card·candidate Owner cardinality는 vote 및 같은 round vote 집합과 대조한다. canonical command digest는 receipt ID를 제외한 명령 의미 필드에서 재계산한다. orphan·중복 result·alias action·raw/prose/secret/control scalar와 mismatch는 migration/open/reconciliation에서 보정 없이 fail-closed한다. S4.2b의 Case/Request 전이, Registry/central authorization, command replay, Manager escalation 및 외부 delivery는 이 capability만으로 활성화되지 않는다.

S4.2b는 `sqlite_durable_direct_conflict_concurrence`로 위 capability의 유일한 `conflict.concur` UoW를 열었다. Completion shared transaction에서 current open Case·AwaitingConflict Request·revision·ordered candidate hash, 현재 Registry Card/Owner/domain, actor ResourceRef 중앙 grant를 시작과 첫 write 직전에 모두 재검증한다. 모든 후보 Owner가 같은 target을 선택한 마지막 vote만 Case `resolved`, Request `ReadyToDispatch(attempt=1)`, receipt graph를 atomic CAS한다. replay도 현재 권한뿐 아니라 resolved Case·round·revision, selected current RouteTarget, Ready route/attempt/trigger/system handling이 receipt와 정확히 맞는지 검증한다. timestamp는 schema와 같은 canonical calendar/offset grammar로 쓰기 전 검증한다. Manager escalation, ticket/wake/delivery, lease/scheduler 및 legacy recovery는 여전히 열지 않는다.

S4.3은 hash-only Case에서 Registry drift cause를 추정하지 않는다. S4.3a는 새 Case용 ordered Candidate Card/Owner/domain/route candidate-only baseline을 별 component에 저장하며, under-claim과 manager/root 관계는 typed Registry 계약이 없으면 저장·추정하지 않는 explicit unavailable이다. S4.3a.1은 requester-only `conflict.open`와 scope-proof Registry snapshot을 시작·write 직전에 확인한 뒤 existing `Received(revision=0)` Request의 Case·baseline·AwaitingConflict Request·under-claim companion evidence graph를 한 Completion transaction에 최초 생성한다. legacy InMemory Conflict-open은 write-through/backfill source가 아니며 baseline 없는 기존 Case도 escalation unavailable이다.

S4.3b sealed escalation evidence는 mutation authority가 아닌 SQLite read-only reader다. 조직 범위의 open Case·AwaitingConflict Request·baseline·ingress companion·direct votes를 validate-only로 읽고, caller가 제공한 ordered raw claim을 ingress claim digest 및 baseline의 ordinal/Card/Owner/domain/route에 exact 증명한다. 이어 current Registry snapshot digest를 vote 판정보다 먼저 비교한다. mismatch는 `CandidateRegistryChanged(snapshot_digest_mismatch)`, 후보 Owner 표결 부족은 `Pending`, 모든 후보 Owner의 상이 target만 `DivergentVotes`다. open Case의 unanimous 또는 부분 합의는 evidence를 추정·보정하지 않고 fail-closed한다. 결과에는 typed ref·digest·round만 있고 raw claim·manager/root은 없다. schema/write/receipt/audit/outbox/action/Manager selection은 이 reader에 없다.

따라서 `conflict.escalate`는 아직 활성화하지 않는다. S4.3c가 ADR0050 중앙 action/role/resource mapping, exact under-claim snapshot, current ResourceRef의 실행 직전 재인가, 사람 승인/HITL 및 immutable receipt·secret-free audit/outbox와 Case escalated·FromDeadlock ManagerItem·Request AwaitingManager의 단일 transaction을 함께 증명하기 전에는 어떤 MCP mutation도 sealed evidence에서 파생되지 않는다.

S4.3c.0은 그 중 graph selection proof만 별 `ConflictEscalationRegistrySnapshot` port로 닫았다. 이는 Conflict-open snapshot이나 legacy Manager graph의 확장이 아니다. composition-owned org scope proof가 있는 한 Registry guard에서 ingress ordered claim의 current Card/Owner/intent/RouteTarget/under-claim, 후보 Owner별 cycle-free manager path, 동일 유일 root, 최대 거리 기준의 유일 nearest common Manager를 확인한다. common Manager가 없을 때만 유일 root fallback을 허용하며 cycle·self-loop·dangling relation·multiple root·selection tie·scope/claim/graph drift는 unavailable이다. `verify_current`는 전체 typed snapshot equality를 다시 확인한다. raw User/path는 외부로 내보내거나 persistence하지 않고 typed ref/digest만 노출한다. 이 계약은 schema/write/action/HITL/receipt/ManagerItem transition을 여전히 만들지 않는다.

manual 재지정은 정확히 `approval.reassign` 중앙 action과 predecessor `approval_item` ResourceRef를 사용하고, 별 `ApprovalReassignmentAuthorization`을 target approver·새 requirement·due_at·policy/authority/evidence ref까지 대조한다. 만료 scan은 묵시적 system 권한으로 실행하지 않는다. 서비스 principal이 `approval.expire` action으로 같은 current ResourceRef의 grant를 받아야 하며, 이 action은 ADR 0050의 중앙 action manifest와 정책에 명시 추가한다. expiry policy는 transaction 안에서 한 번 읽은 DB time으로만 `ReassignExpiredApproval | ApprovalUnavailable` sealed 결과를 만든다. 두 action 모두 current assignment·Request·ResourceRef를 write 직전에 재조회·재인가한다. 권한/정책/time 의존성이 unavailable이면 쓰지 않는다.

lineage는 `(request_id, attempt)` 안에서 predecessor open·current Request `AwaitingApproval` revision, successor의 round+1·awaiting revision+1·같은 org/request/attempt/route/draft, predecessor의 exact supersession evidence와 successor link를 함께 요구한다. `unavailable`은 successor 없이 predecessor를 unavailable로 닫고 같은 Request를 `Failed(approval_unavailable)`로 종결한다. 어느 경우에도 Runtime·Router·Precedent·Authority policy write를 호출하지 않는다. assignment status CAS, Request CAS, lifecycle evidence/result receipt, secret-free audit/outbox intent는 Completion의 typed transaction ownership context 안에서 한 transaction으로 commit한다.

recovery는 schema/row lineage를 read-only로 reconcile한 뒤 committed lifecycle receipt만 복원한다. commit 전 crash는 전부 rollback이므로 새 scan이 현재 open assignment를 다시 평가할 수 있지만, receipt 없는 과거 policy 결정을 추정·backfill·재실행하지 않는다. 외부 전달 소비와 lease는 이 component가 만들지 않는다. DB time은 adapter가 transaction-scoped timezone-aware instant를 제공하는 explicit port이며 SQLite conformance와 PostgreSQL target은 각각 그 값을 독립 검증한다.

이후 순서는 v2 Approval identity migration, durable Approval terminal UoW, Conflict/Manager/WorkTicket transition, transactional outbox·lease/recovery, PostgreSQL migration/cutover다. `request_id=None` legacy 행과 generation 없는 v1 approval receipt는 read-only/non-authoritative이며 자동 replay·backfill authority가 아니다. 이 schema capability는 아직 Approval operation을 실행·이관하거나 Approval과 Completion을 한 transaction으로 묶지 않는다.

## R3 credential-stage parent reservation

secure delivery stage는 actual credential materialization보다 앞서므로, `active|revoked` lifecycle의 `durable_credentials`를 provisional parent로 사용하지 않는다. R3.1은 immutable `Credential Issue Target` reservation을 별 aggregate로 만들고, target의 same semantic command digest와 active credential ID uniqueness를 stage 전에 확인한다. v2 stage fence는 target을 composite FK로 참조하며 `Reserved → StageClaimed → Staged → Committing → Committed | CleanupPending → Cleaned`의 sealed lifecycle만 갖는다.

기존 direct-credential-FK stage fence v1은 migration, backfill, activation 없이 unavailable으로 유지한다. target은 raw secret을 보관하지 않고 semantic stage key, secret hash, approval/resource mirrors와 secret-free delivery reference만 결박한다. R3.2만 target/fence의 exact Staged snapshot을 actual credential, receipt, audit/outbox와 하나의 transaction으로 materialize할 수 있으며, R4는 CleanupPending의 저장된 ref만 abort한다.

## R4 cleanup reconciliation

R4 cleanup은 materialization이나 release의 재실행 경로가 아니다. exact `CleanupPending` target/fence와 그 persisted `delivery_ref`만 대상으로, 외부 abort 전에 same-org target/ref에 결박된 durable cleanup intent와 append-only attempt를 기록한다. abort 성공만 immutable result 및 target/fence 양쪽 `Cleaned` CAS와 함께 확정한다. exception, crash 또는 외부 결과의 모호함은 `CleanupPending`을 유지하므로 재시작은 같은 ref만 다시 abort할 수 있다.

intent·attempt·result의 ref mirror는 composite FK로 결박하고 UPDATE/DELETE 차단 trigger와 canonical UTC-millisecond timestamp 검증으로 이력을 보존한다. raw secret, body, grant는 이 schema에 저장하지 않는다. SQLite 조정은 at-least-once abort recovery이며 external system의 physical exactly-once를 주장하지 않는다.

## R5 credential MCP activation 순서

legacy credential MCP의 direct registry/delivery 도구는 authority가 아니며 public factory는 zero-tool surface로 닫는다. 실제 `issue | list | get | revoke` 공개는 빈 matrix에 머무르지 않고 다음 companion 순서를 모두 충족한 뒤에만 연다: immutable `Credential Scope Binding`이 Target과 같은 transaction에서 credential·Agent Card·Owner·source provenance를 결박하고, scoped issue operation이 current proof·central issue action·HITL·R3 stage/materialize/release를 재검증하며, committed `Credential Scope Projection`이 materialized credential과 binding을 1:1로 고정한다. read와 revoke는 projection 및 current source proof가 있어야 하며, revoke는 approval과 generation/revision CAS를 write 직전에 다시 대조한다.

R5.1은 그 첫 companion만 구현한다. binding open/reservation은 trusted current source snapshot을 required input으로 삼아 credential·Card·Owner/resource fingerprint와 source kind/instance/revision/digest를 모든 stored row에 exact 대조한다. Target과 binding insertion은 하나의 transaction이고, fault 또는 source failure는 둘 다 rollback한다. valid hash를 붙인 out-of-band binding도 current provenance와 다르면 unavailable이며, legacy target/credential/receipt를 binding으로 backfill하지 않는다.

R5.2a는 Target/Fence의 durable transition state machine을 neutral internal core로 모은다. core는 scope source·central authority·MCP·release·abort를 모르며, scoped production bridge와 legacy test-support bridge의 sealed guard만 받는다. scoped guard는 R5.1 binding capability와 current source·principal·`worker_credential.issue` grant·approval action/digest/resource를 claim/pre-external 및 materialization prepare/prewrite의 같은 transaction connection에서 대조한다. core materialization은 release를 호출하지 않고 canonical persisted ref를 secret-free 값으로 반환한다. staged 검증 실패만 exact ref를 `CleanupPending`으로 CAS하며 committed replay 검증 실패는 state를 변경하지 않는다. R4 abort는 여전히 dedicated reconciliation만 수행한다.

R5.2 outer bridge는 core를 공개하지 않는다. path-bound single-use capability가 sealed scoped operations와 R4 readiness를 함께 claim하고 server-side approval acquisition→resolver roundtrip→R5.1 reservation→typed stage→materialize/readback→동일 persisted ref release를 수행한다. raw secret은 최초 stage 필요 상태에만 생성하며 staged/committed replay 또는 ClaimedRecovery에서 새 secret으로 hash를 바꾸지 않는다. release fault와 pre-release current-proof drift는 `ReleasePending`이며 R4 abort를 호출하지 않는다. R4 readiness classifier는 canonical R4/v2 catalog/FK/row validation을 통과한 exact pending pair만 `CleanupRequired`로 표현하고, missing/corrupt/tampered storage는 unavailable이다.

R5.3은 scoped materialization transaction의 mandatory immutable companion인 `Credential Scope Projection`을 추가한다. projection은 `(org_id,target_id)`와 `(org_id,credential_id)`를 각각 하나로 고정하고, Target generation·credential/Card/Owner ResourceRef fingerprint·binding/source provenance·binding/commit timestamp·projection digest를 Binding, actual credential, Committed Target/Fence, receipt와 exact 대조한다. companion persist/readback failure는 credential·receipt·audit·outbox·Target/Fence를 함께 rollback한다. committed read와 outer release는 projection strict validation을 반드시 먼저 통과해야 하며, projection 부재·catalog object 추가/삭제·위조·current source drift는 ref 반환·release·repair/backfill 없이 unavailable이다. transition core는 projection 이름을 모르고 neutral companion hook만 가진다.

R5.4의 read/revoke는 legacy `DurableCredentialRegistry`를 직접 공개하지 않는 별 sealed scoped application capability다. `CredentialReadView`는 credential lifecycle 공개 필드만 가진 secret-free DTO이며 secret/hash·delivery ref·approval evidence·grant·source/projection internals를 DTO·오류·receipt·audit·outbox에 넣지 않는다. list는 organization `worker_credential.read`와 반환 직전 각 current credential ResourceRef 재인가를 모두 요구하고 하나라도 불가하면 partial result 없이 unavailable이다. get/revoke의 존재하지 않는 credential은 neutral `not_found_or_denied`이며, 존재하지만 projection/source/catalog가 없거나 불일치하면 unavailable이다.

Projection은 발급 당시 immutable provenance anchor이고 revoke 뒤에도 유효해야 한다. 따라서 target/receipt/fence/issued generation/owner/commit timestamp는 고정하되 current lifecycle은 동일 generation의 `revision >= 1`, `active|revoked`로 별도 검증한다. revoke는 immutable projection/current source/current ResourceRef의 `worker_credential.revoke` grant와 server-side HITL evidence를 취득·resolver roundtrip한 뒤, 같은 write transaction에서 다시 검증하고 expected generation/revision의 active-row CAS, result receipt, evidence-free procedural audit/outbox를 원자적으로 쓴다. receipt replay도 current source/projection/per-resource reauthorization을 다시 통과해야 하며, exact receipt가 없을 때 이미 revoked 상태를 성공으로 추정하지 않는다. generic registry audit의 evidence ID persistence는 이 path에 재사용하지 않는다.

Revoke receipt exactness는 evidence/digest가 아닌 closed scalar binding으로 기록한다: contract version, fixed action, `(org_id, command_id, attempt)`, credential ID, expected generation/revision 및 server principal의 subject/provider/session identity tuple이다. 그 tuple 하나라도 다르면 conflict와 write 0이며, same tuple replay도 current projection/source/result/current credential 및 revoke grant를 다시 검증한다. 최초 revoke는 provider에서 evidence를 한 번만 취득하고 resolver initial roundtrip 및 write 직전 same-evidence 재확인만 한다; replay는 provider/resolver를 재호출하지 않는다. revoked timestamp는 caller input이 아니라 sealed server clock의 canonical UTC-millisecond 값이다.

R5.5 enabled credential MCP는 legacy zero-tool factory와 별개다. R5.2 issue와 R5.4 read/revoke의 sealed capability·같은 path·org·server principal을 claim 전 모두 검증하고, provisional transport의 constructor/네 decorator/final matrix equality가 모두 성공한 후에만 aggregate와 child capability를 원자적으로 claim한다. capability 경쟁 또는 registration failure는 어느 child도 부분 소진하지 않는다. tool input은 opaque request를 manual parse하여 transport validation의 field/value reflection을 차단하고, success/error 모두 raw secret·hash·delivery ref·approval/evidence·grant·source internals를 반환하지 않는다. 이 boundary는 R4 abort/reconciliation을 호출하지 않는다.

기존 credential, target, receipt에서 Card/Owner provenance를 추정하거나 backfill해 tool을 열지 않는다. partial composition·scope/source drift·legacy unbound row는 도구 등록 0 또는 unavailable이다. R4 abort는 issue handler의 best-effort 처리로 재도입하지 않고 persisted-ref reconciliation으로만 유지한다.

## 결과와 한계

credential MCP가 실제로 열리기 전 선행 조건을 명확히 하고, token 발급·철회와 worker binding을 durable evidence에 결박할 수 있다. 하지만 at-least-once outbox는 외부 채널의 physical exactly-once를 뜻하지 않으며, migration·복구·backup/restore·DR의 실행 증명은 뒤 단계에 남는다.
