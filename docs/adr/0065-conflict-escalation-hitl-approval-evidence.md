# ADR 0065 — Conflict escalation의 HITL 승인 증거 계약 (durable `conflict.escalate`)

- 상태: 채택(Accepted)
- 날짜: 2026-07-22
- 계보: ADR 0046(request-aware Contested 책임 결정·escalation 도메인)이 InMemory 단일 프로세스로 미룬 durable escalation 권한·사람 통제를 P17.9 S4.3c.1로 잇는다. ADR 0050(중앙 Authority·RBAC — §11의 `conflict.escalate` action/role/resource)·ADR 0051(운영 변경 승인 증거 — canonical command digest·resource fingerprint 1:1 결박)·ADR 0052(R5.4 `CredentialApprovalEvidence` 취득/재확인 패턴)·ADR 0044(SQLite Completion UoW)를 재사용하고, S4.3b `SealedEscalationEvidence`·S4.3c.0 `ConflictEscalationRegistrySnapshot`(graph_digest)를 소비 결박한다.
- 적용 범위: durable open ConflictCase를 Manager로 넘기는 `conflict.escalate` 명령의 사람 통제(HITL) 승인 증거 — 값 객체 shape, canonical command/cause digest, 취득 provider·재확인 resolver 포트, 순수 검증 계약, 승인자·만료·취소·재승인의 표현 위치.
- 제외 범위: 동시성/장애 게이트(S4.3c.4), reconciliation 게이트(S4.3d), durable Manager 처분(S4.4 — FromDeadlock Item을 Assign/Dismiss로 소비). 파일럿 조직의 구체 승인자 신원·실 HITL UI·durable policy epoch·다중 인스턴스는 여전히 후속이다. escalation receipt graph 스키마의 지속 shape는 §8에서, start/write 직전 재인가·재확인과 Case/ManagerItem/Request 전이를 한 transaction에 쓰는 UoW(S4.3c.3) 계약은 §9에서, 그 committed 결과의 cross-aggregate 정합을 read-only로 증명하는 reconciliation 게이트(S4.3d) 계약은 §10에서, S4.4 durable Manager 처분의 FromDeadlock 소비 계약은 §11에서, S4.1 command receipt(manager.assign_owner/dismiss·work_ticket.create) ⟺ target aggregate·Request 상태의 read-only reconciliation 게이트(S4.6) 계약은 §12에서 보강했다(동시성·장애 행렬 증명은 c.4 밖, repair/wake/lease는 여전히 S4.4~S5 밖).

## 맥락 — escalation 자동 seal에서 durable 사람 통제 처분으로

ADR 0046(InMemory S1~S6)에서 escalation은 자동이었다. 마지막 상이 표결이 `DivergentVotes`/`CandidateRegistryChanged` claim을 seal하면 애플리케이션이 곧바로 FromDeadlock ManagerItem을 만들고 Case를 `escalated`로, Request를 `AwaitingManager`로 옮겼다. 그 ADR은 durable 내구성·권한·RBAC를 "P17.9 또는 후속 ADR"로 명시 이관했다.

P17.9 S4.3은 이 경계를 durable하게 다시 연다. S4.3a는 escalation baseline을, S4.3b는 표결 없는 read-only `SealedEscalationEvidence`(`DivergentVotes | CandidateRegistryChanged`)를, S4.3c.0은 graph-aware `ConflictEscalationRegistrySnapshot`(유일 nearest common Manager 또는 유일 root와 `graph_digest`)을 닫았다. 남은 것은 이 두 sealed 산물을 소비해 실제로 Case를 escalated로 전이하는 durable 명령이다.

그런데 durable 전이(Case `escalated`·FromDeadlock ManagerItem·Request `AwaitingManager`)는 되돌리기 어려운 운영 처분이다. ADR 0051이 `session.end`·`hitl.write`·`card.transfer_owner`에 "별도 운영 변경 승인 gate의 현재 허용·증적 참조"를 요구하고 "기존 HITL 토글을 이 승인으로 해석하지 않는다"고 못박은 것과 같은 결로, escalation도 중앙 grant만으로 열어선 안 된다. 사람 승인 증거가 필요하고, 그 증거는 이 명령·이 Case·이 escalation 원인·이 graph 선택에 exact 결박돼야 다른 명령·다른 Case·낡은 원인으로 재사용되지 않는다.

이 계약은 되돌리기 어려운 새 결정 — 승인자·만료·취소·재승인 lifecycle과 toggle 불인정 — 을 담으므로, action/role/resource만 다루는 ADR 0050 §11과 분리해 별 ADR로 둔다.

## 결정

### 1. ADR 경계 — ADR 0050 §11(action) + 신규 ADR 0065(evidence)

`conflict.escalate` action·role·resource mapping은 ADR 0050 §11 보강으로 충분하다. `conflict.open` 선례(a.0)가 이미 같은 층(중앙 action manifest·role hard-limit·resource kind·resolver re-read)에 있기 때문이다.

그러나 HITL 승인 증거 계약은 별 ADR(이 문서)로 둔다.

- **ADR 0046은 스스로 경계를 그었다.** durable·권한·RBAC를 후속 ADR로 명시 이관했으므로, 그 escalation 도메인 ADR(이미 1,400줄 넘음)을 다시 열어 durable HITL 증거를 끼우면 그 경계가 흐려진다.
- **승인자·lifecycle·toggle 불인정은 되돌리기 어려운 새 결정**이라 프로젝트 규율상 자기 ADR을 갖는다.
- **형제 배치가 일관된다.** ADR 0051(운영 승인 증거)·ADR 0052(R5.4 credential 승인 증거)는 각 명령 도메인의 승인 증거 계약을 자기 ADR로 둔다. escalation 승인 증거도 같은 자리에 둔다. RBAC(0050)와 evidence lifecycle(0065)은 다른 관심사다.

### 2. role — operator 명령 + 별도 HITL 승인자 (R5.4 동형 2층)

`conflict.escalate`는 R5.4(중앙 grant AND HITL)와 동형인 AND 2층이다.

- **1층(중앙 RBAC grant):** 명령 주체 role은 **operator**다(ADR 0050 §11에서 hard-limit). escalation은 owner concurrence(`conflict.concur`)도 Manager 처분(`manager.act`)도 아닌 운영 human-control 처분이다. Manager는 escalated Item을 *받아* `manager.act`로 처분하는 downstream 주체이지 자기 escalation을 개시하지 않는다.
- **2층(HITL 승인자):** 명령을 개시하는 operator와 별개로, 사람 승인자의 sealed 증거가 필요하다. 승인자의 구체 신원·조직 그래프 위치는 evidence store가 쥐고 resolver가 current로 증명하며, 중앙 RBAC layer에 승인자 role을 hard-code하지 않는다. 파일럿 조직의 구체 승인자 신원은 외부 결정(차단 아님)이고, 타입/정책 스냅샷 수준에서 승인자는 "사람 governance subject"다(ADR 0017 — owner/사람은 governance 주체). 두 층은 AND이므로, operator grant만으로도, 승인 증거만으로도 escalation은 열리지 않는다.

이 분리는 직무 분리(operator 개시 · 사람 승인 · Manager 처분)를 낳고, ADR 0050 §5의 "admin·operator·auditor는 합치지 않는다" 정신과 정합한다.

### 3. 만료·취소·재승인 — 단일 action + resolver current-snapshot (별 중앙 action 없음)

만료·취소·재승인을 위해 `conflict.escalate.expire` 같은 별 중앙 action을 만들지 않는다. 단일 `conflict.escalate` action + 승인 증거의 current-snapshot resolver로 표현한다.

- **R5.4 선례:** `CredentialApprovalEvidence`는 상태 필드 없이, `CurrentCredentialApprovalEvidenceResolver.resolve_credential_approval_evidence(org_id, evidence_id)`가 current 유효 증거 또는 `None`을 돌려주고, 취득·write 직전 모두 `current == evidence`로 same-evidence를 재확인한다. **만료·취소 = resolver가 `None`(또는 비일치)을 돌려줌**, **재승인 = 새 `evidence_id`가 옛 것을 대체**. 별 중앙 action은 없다.
- **ADR 0050 §6 정합:** 별 action은 manifest를 부풀리고 "한 escalation을 한 명령·한 authorizer 경계로"라는 규율을 깬다. 만료·취소·재승인은 *승인 증거의 lifecycle 상태*이지 별개 RBAC 명령이 아니다.
- **c.1은 shape만.** 실제 평가(취득·재확인)·transaction은 S4.3c.3이다. c.1은 증거 값 객체와 포트 시그니처, 순수 검증 계약(취득 1회·same-evidence 재확인·replay 재취득 0)까지만 못박는다.

### 4. canonical digest — conflict-escalation-domain-local 신설 (durable_credentials generic import 아님)

digest·fingerprint helper는 `durable_credentials`에서 import하지 않고 conflict-escalation 도메인에 신설한다.

- **도메인 결합 회피.** escalation 승인 증거가 `durable_credentials`(무관한 credential 도메인)에 import 의존하면 bounded context가 새어 든다. 헥사고날/DDD 상 두 컨텍스트를 분리한다.
- **신설 지점:** `canonical_escalate_command_digest(...)`(action·resource fingerprint·command에 결박)와 `escalation_cause_digest(...)`(S4.3b `SealedEscalationEvidence`를 canonical 결박)는 이 도메인이 소유한다. resource fingerprint는 `ResourceRef` 4필드만 hash하는 도메인 중립 계산이지만, cross-domain import를 피하려 같은 canonical 형태를 도메인-local helper로 재현한다(사소한 4필드 hash 중복이 import 결합보다 싸다).
- **graph selection은 재사용(신설 아님).** `graph_selection_digest`는 S4.3c.0 `ConflictEscalationRegistrySnapshot.graph_digest`를 그대로 결박한다(manager/root 선택이 이미 그 digest 안). 새 digest 함수가 아니라 field 결박이다.

### 5. sealed 증거 shape — granted 스냅샷 하나 (lifecycle union은 c.2/c.3로 이월)

c.1이 못박는 것은 **소비 가능한 granted 증거 값 객체 하나**다. 만료·취소·재승인 상태의 full union은 c.1이 만들지 않는다(소비 최소 결박).

- **`ConflictEscalationApprovalEvidence`(frozen 값 객체)**: `evidence_id`·`status: Literal["granted"]`·`action: Literal["conflict.escalate"]`·`command_digest`(64)·`resource_fingerprint`(64)·`escalation_cause_digest`(64)·`graph_selection_digest`(64). `CredentialApprovalEvidence`("이미 검증된 사람 승인 증거의 secret-free snapshot")의 escalation 판이다.
- **`status: Literal["granted"]`의 역할(§7 ①과 결합):** 답변 HITL toggle과 구분되는 자기 서술 타입이 되고, resolver가 잘못 낡은 증거를 돌려줘도 값 수준에서 걸러지는 방어선을 준다. c.1은 `granted` 변이만 정의한다. `expired`/`cancelled`/`superseded` 같은 terminal 상태·timestamp·supersession chain은 S4.3c.2 schema·S4.3c.3 UoW가 필요할 때 여는 확장이지 c.1이 미리 짓지 않는다.
- **3중 결박이 핵심 신설:** `command_digest`·`resource_fingerprint`(R5.4 재사용 결)에 더해 `escalation_cause_digest`·`graph_selection_digest`가 escalation 특유의 결박이다. 증거가 *이* 명령·*이* conflict_case·*이* escalation 원인(S4.3b sealed evidence)·*이* graph 선택(S4.3c.0 snapshot)에 exact 묶여, 낡은 원인·다른 Case·drift된 graph로 재사용되지 못한다.

### 6. 취득·재확인 — R5.4 패턴 재사용 (취득 1회·same-evidence 재확인·replay 재취득 0)

- **`EscalationApprovalProvider`(취득 포트):** `acquire_escalate_approval(principal, action, resource, command_digest) -> ConflictEscalationApprovalEvidence`. `CredentialRevokeApprovalProvider.acquire_revoke_approval`의 escalation 판. 취득은 명령 경로에서 **정확히 1회**다.
- **`CurrentEscalationApprovalEvidenceResolver`(재확인 포트):** `resolve_escalation_approval_evidence(*, org_id, evidence_id) -> ConflictEscalationApprovalEvidence | None`. `CurrentCredentialApprovalEvidenceResolver`의 escalation 판. 취득 직후와 write 직전 모두 same-evidence(`current == evidence`)를 재확인한다.
- **replay 재취득 0:** 같은 명령의 replay는 승인을 재취득하지 않고, current scope·중앙 authorization·same-evidence만 다시 검증한다(R5.4 `_current_access` 정신). 사람 승인은 한 번만 요구하고, 재시도가 새 사람 승인을 강요하지 않는다.

### 7. toggle 불인정 강제 — 타입 분리 + action 분리 + no-import 가드

기존 답변 HITL toggle(ADR 0025 `HitlToggleMap`)은 이 escalation 승인 증거가 **아니다**. ADR 0051이 운영 승인에서 toggle을 배제한 것과 같다. 세 겹으로 강제한다.

1. **타입 분리:** `ConflictEscalationApprovalEvidence`는 `HitlToggleMap`·`Answer.mode`와 공유 base 없는 독립 frozen 타입이다. `status: Literal["granted"]` 판별자가 답변 mode toggle과의 혼동을 막는다.
2. **action 분리:** `conflict.escalate`는 답변 흐름 toggle(`hitl.read`/`hitl.write`)과 다른 중앙 action이다. escalation 승인 증거는 `hitl.write` 운영 승인도 아니다.
3. **no-import 가드 테스트:** 신규 escalation 증거 모듈이 `hitl`(toggle) 모듈을 import하지 않음을 AST로 단언한다. `test_p17_conflict_module은_Router_Precedent_ComplementEdge를_import하지않는다`의 기존 패턴을 재사용한다.

의미 backing: provider·resolver 포트는 HITL toggle을 승인 원천으로 받지도 조회하지도 않는다. resolver는 escalation 승인 증거 store만 읽는다.

### 8. c.2 escalation receipt graph 스키마 — receipt-parent 허브 + sealed 증거 recompute (2026-07-22 보강)

c.1이 값 객체·포트·순수 검증(shape)을 닫았으므로, c.2는 그 계약이 durable하게 어떻게 기록되는지를 별 versioned component `durable_conflict_escalation_receipts_v1`로 설치한다. 새 되돌리기 어려운 도메인 결정은 없고(권한·lifecycle·toggle 불인정은 §1~§7이 이미 정함), c.2는 §5 4중 결박·S4.3b sealed cause·S4.3c.0 graph selection의 지속 shape만 못박으므로 자기 ADR 없이 이 ADR 보강으로 둔다(S4.2a schema가 자기 ADR 없이 ADR 0046/0051 계보로 처리된 선례와 같다).

- **receipt가 parent다(R1.0 판, S4.2a 판 아님).** S4.3b·c.0은 read-only라 escalation 명령 전에 어떤 durable row도 남기지 않는다. sealed cause·graph selection은 명령 시점에 계산·봉인돼 c.3의 한 transaction으로 receipt와 *함께* 처음 쓰인다. S4.2a votes처럼 명령 이전에 누적되는 독립 lifecycle이 없으므로, evidence를 parent로 두면 실제 생성 순서를 뒤집는다. escalation 하나 = receipt 하나이고 sealed 산물은 그 명령의 속성이다 — 이는 receipt가 source-scope proof digest를 자기 컬럼으로 지니는 R1.0 판과 동형이다. child(sealed evidence·result projection·audit·outbox intent)는 R1.0의 `(org_id, receipt_id)` same-org composite FK로 receipt에 1:1 결박해 cross-org row stitching을 DB 수준(PRAGMA foreign_keys=ON)에서 막는다.
- **secret-free 컬럼만.** opaque typed ref(`kind:<lowercase SHA-256>`)·64-hex digest·canonical UTC timestamp·범위 정수만 저장한다(S4.1 규율). raw claim·사유 원문·secret은 없다. c.0 `manager_subject_ref`(nullable)·`root_subject_ref`는 *digest가 아닌 typed subject ref*로 저장한다 — baseline이 `candidate_owner_subject_ref`를 저장하는 것과 같은 secret-free 문법이고, c.3가 선택 target을 FromDeadlock ManagerItem `manager_subject_id`에 mirror하려면 봉인된 선택 대상이 durable해야 하기 때문이다. raw User ID는 여전히 저장하지 않는다.
- **`escalation_cause_digest`만 recompute(자기 정합), 나머지 digest는 store+mirror.** sealed cause 필드(kind·org/conflict/request ref·revision·round·candidate_snapshot/baseline/candidate_claim/vote_set sha256·variant)를 evidence row에 저장하고 c.1과 같은 canonical 필드 순서로 `escalation_cause_digest`를 재계산해 receipt 값과 일치를 강제한다(S4.2a `command_digest`·baseline `baseline_sha256` recompute 규율). `graph_selection_digest`(c.0 소유)·`command_digest`·`resource_fingerprint`(c.1/c.3 소유, 원본 command·ResourceRef 미저장)는 재계산하지 않고 64-hex·receipt↔evidence mirror만 강제한다. `durable_credentials`·c.1 domain 값 객체를 import하지 않고 canonical hash를 도메인-local로 재현한다(§4 no-import 규율).
- **validate-only·write API 0.** open/reconcile은 검증만 하고 repair 0이다(S4.1 판). c.3가 소비할 쓰기 표면은 c.2가 예비하지 않는다 — S4.1/S4.2a/S4.3a 전 선례가 schema는 validate/migrate/open/reconcile만 노출하고 write는 UoW 모듈(S4.2b 동형)이 소유하며 schema validate를 호출할 뿐이다(inactive port 아님, 스키마만). fault-atomic migration·manifest 없는 partial/DDL drift fail-closed·S4.1/S4.2/S4.3a DDL 무변경은 전 선례 그대로다.

### 9. c.3 escalation UoW — 세 aggregate·receipt graph 원자 전이 계약 (2026-07-22 보강)

c.1(§1~7)이 승인 증거 shape·포트·순수 검증을, c.2(§8)가 receipt graph 지속 shape를 닫았으므로, c.3는 그 둘을 소비해 **하나의 `conflict.escalate` 명령을 한 SQLite transaction으로 원자 전이**하는 UoW를 연다. 새 되돌리기 어려운 도메인 결정은 없다 — 권한 중앙·HITL·toggle 불인정·receipt-parent는 §1~§8이 이미 정했고, c.3는 그 계약들의 조립·쓰기 순서·terminal Case에서의 replay 의미만 확정하므로 자기 ADR 없이 이 보강으로 둔다(S4.2b UoW가 자기 ADR 없이 처리된 선례와 같다). tdd 소유 모듈은 `sqlite_durable_conflict_escalation_uow.py`(S4.2b `sqlite_durable_direct_conflict_concurrence` 동형).

- **Q1 — transaction seam·조립(Completion shared connection).** S4.2b와 동일하게 c.3는 `SqliteQuestionCompletionUnitOfWork.durable_transaction()`이 주는 `SqliteCompletionTransaction` 하나로 세 durable capability를 한 파일·한 `BEGIN IMMEDIATE`에서 쓴다. 그러려면 **S4.1 linked aggregates(Case·ManagerItem)·c.2 receipt graph·Completion question_requests가 같은 SQLite 파일**에 있어야 한다. 생성 시(transaction 밖) `validate_component`로 parent 집합을 read-only 검증하고 — c.2 receipt graph(→escalation baseline S4.3a로 cascade)·S4.1 linked aggregates·Completion — 하나라도 실패하면 `Unavailable`로 fail-closed한다. 명령 진입 시 `begin_immediate` 직후 같은 parent를 `validate_component_in_transaction`으로 재검증한다(replay가 open-time 검증을 신뢰하지 않는 S4.2b 규율 — companion row가 명령 사이에 오염될 수 있다).

- **Q2 — start vs prewrite 재확인 분담(모두 transaction 안, 각 2회).** ① **중앙 재인가**: `conflict.escalate` grant + `verify`를 start 1회·prewrite 1회. `ConflictEscalateCaseResolver`가 durable open Case + AwaitingConflict Request를 증명해야 grant가 난다(0050 §11 봉인). ② **HITL 재확인**: 취득은 `acquire_escalation_approval`로 **정확히 1회**(provider fetch — 이미 봉인된 승인 snapshot 조회이지 사람 대기 아님) — 그 안에 resolver same-evidence 재확인 1회가 포함되고, prewrite에서 `reconfirm_escalation_approval`로 same-evidence + 4결박을 1회 더. ③ **sealed cause 재검증 = S4.3b reader 재실행**(caller digest 대조 아님): c.3가 `SealedEscalationCauseReader` 포트(S4.3b `DurableConflictEscalationEvidenceReader`)를 주입받아 start에 `cause0`를, prewrite에 `cause1`을 재도출하고 `cause1 == cause0` + `escalation_cause_digest(cause1) == evidence.escalation_cause_digest`를 강제한다(`Pending`은 escalatable 아님 — fail-closed). server-derived proof라 caller가 원인을 자기보고하지 않는다. ordered `claims`는 caller 입력이되 reader가 durable ingress digest에 pin한다. ④ **graph snapshot**: prewrite에 `verify_current(graph0, claims)`로 Registry drift를 닫는다. `begin_immediate`의 RESERVED가 durable vote를 얼리므로 cause0/cause1은 동일 durable 상태를 읽고, 차이는 오직 SQLite lock 밖 in-memory Registry drift에서 온다(cause 재도출·graph verify_current가 그 drift를 둘 다 잡는다).

- **Q3 — replay·경합 의미론(escalation은 terminal — S4.2b와 갈린다).** 세 aggregate 전이와 receipt가 한 transaction이므로 **`receipt(org,conflict_id) 존재 ⟺ Case escalated`(원자 불변식)**. `begin_immediate` 직후 Case를 읽고 `(org_id, conflict_id)`로 c.2 receipt를 조회해 분기한다.
  - **같은 canonical command 재시도(receipt 존재·`command_digest` 일치):** 저장 result를 그대로 돌려주는 멱등 경로. HITL 재취득 0·**중앙 재인가 재실행 안 함·S4.3b cause 재도출 안 함** — Case가 이미 `escalated`라 `ConflictEscalateCaseResolver`(open 요구)도 S4.3b reader(open Case + AwaitingConflict 요구)도 구조적으로 통과 불가이기 때문이다. 재검증은 (a) receipt가 이 canonical command에 결박(`command_digest`·`resource_fingerprint`), (b) 세 aggregate가 커밋 후 정확한 상태(Case `escalated`@awaiting_revision·FromDeadlock ManagerItem open·Request `AwaitingManager(contested)`)임을 확인하고 stored result 반환. **불변한 receipt가 authorization+approval proof**다(S4.2b가 replay에서 immutable projection을 재검증하고 vote를 재도출하지 않는 것과 동형). 이는 §6의 "replay 중앙 authorization 재검증"을 terminal Case에 맞게 정련한 것으로, concur의 Case는 vote 사이 `open`을 유지해 재인가가 가능하지만 escalate의 Case는 `escalated`로 넘어가 재인가 자체가 성립하지 않는다 — Authority 중앙 불변식은 authority가 어디서 선언되는가의 규율이지 멱등 재시도마다 authorize를 재실행하라는 요구가 아니다. `approval_evidence_id`는 receipt에 `evidence:<sha256>`로 봉인(c.2 DDL 고정·secret-free)돼 원본 id를 복원할 수 없으므로 replay는 resolver same-evidence reconfirm을 수행하지 않는다.
  - **이미 escalated Case에 다른 command(receipt 존재·`command_digest` 불일치):** conflict — `UNIQUE(org_id, conflict_id)` 의미론(receipt 하나/Conflict)이자 Case가 open 아님. write 0.
  - **receipt 없음:** fresh 경로. Case가 open 아니면(resolved/escalated) 중앙 재인가·Case read가 거부(write 0).
  - **독립 리뷰 교차검토 결과·경계 명시(2026-07-22·APPROVE):** ① 이 replay 논거는 **실 `central_authorizer`가 Case-state-aware(`ConflictEscalateCaseResolver`가 open만 증명)라는 가정**에 선다 — 실 resolver 와이어(P17.9 후속 배선) 시 이 가정을 명세·테스트로 못박아야 한다. ② replay 반환은 typed digest ref·정수뿐이라 노출 불변식 위반이 아니며(메타데이터 등급), cross-org는 org-필터 조회+digest preimage의 org 결박+c.2 `UNIQUE(org_id,conflict_id)`·same-org FK 3중으로 차단, digest는 서버측 재계산이라 위조 불가. ③ replay 멱등 창은 **의도적으로 downstream 미진행에 결박**된다 — S4.4가 ManagerItem을 소비한 뒤 도착한 escalate replay는 stored result가 아니라 Unavailable이며, 불변 receipt는 durable proof로 존속한다(escalate 재시도는 즉시 post-escalation 상태에서만 의미가 있고, 그 뒤는 조회의 문제지 명령 멱등의 문제가 아니다). ④ c.3의 CAS 경합 테스트는 단일 커넥션 동일-transaction 시뮬레이션까지다 — 실 다중 커넥션 32-way 경합은 c.4가 검증한다.

- **Q4 — 세 aggregate CAS·ManagerItem shape.** ① **Case**: `UPDATE durable_linked_conflict_cases SET status='escalated' WHERE conflict_id=? AND status='open' AND awaiting_revision=?` rowcount==1(expected awaiting_revision = `cause.awaiting_revision`, escalation은 awaiting_revision 불변). ② **Request**: AwaitingConflict→AwaitingManager를 `QuestionRequest.transition` + `compare_and_set_question_request`로. **revision 규약은 S4.3b/conflict-open ingress를 따른다 — AwaitingConflict `request.revision == Case.awaiting_revision + 1`**(ingress가 Received rev0→AwaitingConflict rev1, Case.awaiting_revision=0으로 씀). escalation은 revision을 `awaiting_revision + 2`로 올린다. 목표 상태 `AwaitingManager(item_id=manager_item_ref, public_kind="contested", route=None, attempt=None, handling=HandlingAssignment(kind="manager_item", ref=manager_item_ref))` — ADR 0042 전이표가 AwaitingConflict→AwaitingManager를 `contested`로만 허용한다. ③ **FromDeadlock ManagerItem 생성(S4.1 스키마 행·c.3는 생성까지만)**: `durable_linked_manager_items`에 `source_kind="deadlock"`·`source_ref=source:<sha256(conflict_id)>`·`manager_subject_id=<선택 target>`·`status="open"`·`awaiting_revision=cause.awaiting_revision`·`request_id UNIQUE`(멱등 2차 앵커). **선택 target은 c.0/c.2 result projection과 결박**: `graph0.manager_subject_ref`가 있으면 그 값(result_kind `escalated_to_manager`), 없으면 `graph0.root_subject_ref`(result_kind `escalated_to_root`). ManagerItem `manager_subject_id` == c.2 `result_projection.target_subject_ref` == AwaitingManager가 가리키는 처분 주체. S4.4가 나중에 여는 Manager Assign/Dismiss와 소유권 충돌 없게 c.3는 status="open" Item *생성*까지만 하고 처분(전이·evidence proof)은 열지 않는다. c.3는 S4.1 `durable_linked_command_receipts`/audit/outbox는 **쓰지 않는다** — escalation의 canonical receipt/audit/outbox는 c.2 graph가 소유한다(S4.2b가 자기 receipt graph만 쓰고 S4.1 command_receipts를 건드리지 않은 판).

- **Q5 — fault points·부분 쓰기 0(all-or-nothing).** 모든 write가 한 `begin_immediate` transaction 안이라 임의 지점 예외 → rollback → partial write 0. c.4가 각 seam에 예외를 주입하도록 `fault_injector: Callable[[str], None] | None`을 노출한다(S4.2b `after_vote`/`after_receipt_graph`/`after_case_request` 판). write 순서는 FK(receipt parent → child, PRAGMA foreign_keys=ON)를 지켜: `receipt → evidence → result_projection → audit_intent → outbox_intent`(c.2, Case 아직 open) → `manager_item`(S4.1) → **Case CAS escalated** → **Request CAS AwaitingManager** → commit. fault point는 각 write 뒤(`after_receipt`·`after_evidence`·`after_result_projection`·`after_audit_intent`·`after_outbox_intent`·`after_manager_item`·`after_case_escalated`·`after_request_awaiting_manager`). c.2 receipt는 Case가 open일 때 삽입되지만 c.2 row 검증이 status가 아닌 `awaiting_revision` mirror만 보므로 escalated 후에도 유효하다.

- **전이 ≠ 기록(불변식 보존).** 세 aggregate 전이(Case status·FromDeadlock ManagerItem·Request state — 도메인)와 c.2 receipt/evidence/projection/audit/outbox(기록)는 원자성을 위해 한 transaction에 함께 쓰이되 개념은 구분된다 — receipt는 전이가 아니라 전이의 불변 감사 기록이다. **미아 없음**: escalation은 deadlock/drift된 Conflict를 사람(Manager/root)에게 durable하게 닿게 하는 종착 보존 경로다(ManagerItem은 반드시 유일 manager 또는 root 큐로 간다). **Authority 중앙**: 명령 role(operator)·Manager/root 선택(c.0 graph)·open Case 증명은 중앙 Registry/정책/resolver에서만 온다. **노출 불변식**: 저장은 typed ref·digest·범위 정수·canonical timestamp뿐, raw claim·Manager/root User·원문 0.

### 10. S4.3d escalation reconciliation gate — committed cross-aggregate 정합의 read-only 증명 (2026-07-22 보강)

c.2(§8)가 receipt graph 지속 shape를, c.3(§9)가 세 aggregate·receipt graph 원자 전이를 닫았으므로, S4.3d는 그 커밋된 결과가 실제로 정합함을 **명령·transaction·repair 없이 read-only로 증명하는 게이트**를 연다. 새 되돌리기 어려운 도메인 결정은 없다 — 검증할 불변식(`receipt(org,conflict_id) 존재 ⟺ Case escalated`·세 aggregate 결박)은 §8/§9가 이미 선언했고, S4.3d는 그것을 읽어 대조할 뿐이다. 따라서 자기 ADR 없이 이 보강으로 둔다(S4.2 reconciliation·S3.2d one-shot reconciliation이 자기 ADR 없이 처리된 선례와 같다). 유일하게 새로 못박는 판단은 **"정합 vs 정상 downstream 진행(S4.4 처분)"의 경계**(Q1③·아래 Q3)로, S4.4·미래 독자가 계약을 오해하지 않도록 여기 명문화한다. tdd 소유 모듈은 신규 `sqlite_durable_conflict_escalation_reconciliation.py`(read-only sibling — c.3 UoW와 같은 세 validator를 주입받되 write 표면 0).

- **Q1 — 증분 경계(c.2·c.3와 겹치지 않는 S4.3d 고유 검증).** c.2 reconcile은 receipt graph *내부* 1:1(receipt↔evidence↔projection↔audit↔outbox)·digest 자기정합·receipt→Case/Request *identity lineage*(org·request_id·awaiting_revision·candidate_set_sha256)까지만 보고 **Case.status·ManagerItem·Request.state는 보지 않는다**. c.3 replay는 그 세 aggregate 정합을 *한 conflict에 한해·명령 gated·write transaction 안에서·처분 이전(pre-disposal)만* 검증한다. S4.3d의 고유 증분은 **org-scope 전수 sweep으로, 명령 없이, downstream 진행을 허용하며** 다음 cross-aggregate 정합을 증명하는 것이다.
  - **① `receipt ⟺ Case escalated`(양방향·영구).** (forward) 모든 committed receipt(org,conflict_id) → 그 Case가 존재·status=`escalated`(receipt 있는데 Case가 escalated 아님 = 손상). (backward) 모든 status=`escalated` Case(org) → 정확히 하나의 receipt(org,conflict_id) 존재(escalated Case인데 receipt 없음 = 손상·미아). Case escalation은 terminal이라 이 결박은 downstream 진행과 무관하게 영구다.
  - **② receipt ↔ FromDeadlock ManagerItem(존재+identity 결박·status-tolerant).** receipt.request_id로 ManagerItem(request_id UNIQUE) 존재 + org 일치·request_id 일치·awaiting_revision == receipt.awaiting_revision·source_kind=`deadlock`·source_ref == `source:sha256(conflict_id)`·manager_subject_id == result_projection.target_subject_ref. **status ∈ {open, resolved, dismissed} 전부 유효**(open=처분 전·resolved/dismissed=S4.4 처분 후). ManagerItem 부재 = 손상(미아 없음 — escalation은 반드시 durable Manager 큐에 닿는다).
  - **③ receipt ↔ Request(결박·downstream 진행 tolerant — 핵심 난제).** Request 존재·org 일치·`revision >= Case.awaiting_revision + 2`(escalation 전이 floor — 진행은 오직 revision을 올린다). **판별자는 ManagerItem.status다**(S4.3d가 직접 읽는 authoritative "처분 발생 여부" 신호 — S4.4 mechanics를 추정하지 않는다): **ManagerItem.status=`open`(미처분)** → Request는 정확한 escalation resting shape여야 한다 — `AwaitingManager`·item_id == manager_item_ref·public_kind=`contested`·route None·attempt None·handling(kind=`manager_item`, ref==manager_item_ref)·`revision == awaiting_revision + 2`. **ManagerItem.status ∈ {resolved, dismissed}(S4.4 처분됨)** → Request는 진행했을 수 있다 — `revision >= awaiting_revision + 2`·org 일치만 요구하고, Request가 *아직* `AwaitingManager`면 escalation 결박(item_id·contested·route/attempt None·handling)이 여전히 성립해야 한다. 그 너머의 진행 상태는 손상이 아니다.
  - **④ escalated Case의 부분 companion 결손은 ①~③에 포섭된다.** escalated Case는 backward(①)로 receipt를, receipt는 forward(②③)로 ManagerItem·Request를 요구하므로, 셋 중 하나라도 결손이면 위 검증이 위반으로 잡는다.

- **Q2 — 모듈 배치(별 모듈 신설, c.2 확장 아님).** cross-aggregate reconciler는 receipt graph(c.2)·S4.1 linked aggregates·Completion question_requests를 **가로질러 읽는다**. c.2 schema 모듈에 이를 넣으면 c.2가 S4.1 ManagerItem·Request state에 의존하게 돼(c.2는 Case를 identity FK parent로만 알고 status·ManagerItem·Request state를 모른다) 컴포넌트 소유권이 샌다. 선례도 분리다 — S3.2d approval reconciliation은 schema(`sqlite_durable_approval_lifecycle.py`)와 별 모듈(`sqlite_durable_approval_reconciliation.py`)이다. c.3 UoW가 이미 세 validator를 주입받는 자리이므로, reconciler는 그 옆 read-only sibling으로 둔다.

- **Q3 — report·fail-closed·org-scope.** report는 schema-reconcile 선례(`...ReconciliationReport(capable, detail, ...manifest_present)`)를 계승하되 진단용 **violation 리스트**를 더한다. `capable == True ⟺ org-scope violation 0`(fail-closed 이진 게이트). violation은 typed(`conflict_id`=stored typed ref·`kind`∈{`receipt_without_escalated_case`, `escalated_case_without_receipt`, `manager_item_missing_or_mismatched`, `request_state_inconsistent`, `escalation_capability_uncertain`}·`detail`). **capability-우선 fail-closed:** c.2 receipt graph·S4.1 linked aggregates·Completion을 org_id와 함께 먼저 read-only validate — 하나라도 실패하면 어떤 row도 신뢰 못하므로 즉시 `capable=False`·`escalation_capability_uncertain` 단일 violation으로 닫고 row 열거는 하지 않는다(전역 corrupt catalog는 모두에게 fail-closed·S3.2d `_require_capability`·c.2 판). capability가 서면 org-필터 row를 전수 열거하고 발견한 모든 cross-aggregate violation을 담는다(첫 위반에서 멈추지 않음 — 운영자가 손상 전모를 본다). **org-scope 경계:** row 열거·cross-aggregate 조회는 org-필터(WHERE org_id=?), catalog/manifest/FK/schema 검사는 org-무필터(전역) — c.2/S4.1 `_validate_rows` 판 그대로("전역 corrupt catalog는 거부, 타 조직 row는 scoped reconciliation 동안 opaque"). **violation 시 닫힘:** S4.3d는 보고만 한다 — repair·write·전이 0. 게이트 의미론은 "`capable=False`면 호출자(운영 도구·후속 orchestrator)가 명령 경로를 열지 않는다"이며, S4.3d 자신은 c.2처럼 열 connection handle을 노출하지 않는다(세 aggregate를 한 handle로 "여는" 대상이 아니다). c.2의 `open_...`/`validate_..._connection`을 새로 만들지 않는다 — S4.3d를 in-transaction validator로 주입할 downstream이 없다(마지막 게이트).

- **Q4 — one-shot 경계.** 입력은 `db_path`·`org_id: str | None = None`뿐이다(read-only 검증이라 S3.2d와 달리 principal·authorizer·factory 불요 — dispatch가 없다). cursor·scheduler·lease·wake·delivery 0. 자기 read-only(`mode=ro`) connection을 열고 `PRAGMA foreign_keys=ON` 뒤 **하나의 deferred read transaction(`BEGIN`…`COMMIT`)으로 전 sweep을 한 snapshot에 묶어**(receipt·Case·ManagerItem·Request가 서로 다른 시점을 보지 않게 — cross-aggregate 정합에 필수) 열거·검증·close한다. `org_id=None`이면 전 조직 전역 sweep. typed Request state 디코드는 `_select_question_request_no_commit(connection, request_id)`(sqlite_stores — approval lifecycle reconcile이 bare connection에서 쓰는 승인된 read-only 경로)로 하고, 나머지(receipt·Case·ManagerItem)는 raw SQL로 읽는다. `source:sha256(conflict_id)`·`manager:…` 같은 typed ref 재현은 도메인-local 1줄 helper(`f"source:{sha256(conflict_id)}"`)로 하고 c.3/S4.1 값 객체를 import하지 않는다(§4/§8 no-import 규율).

- **downstream 진행 vs 손상의 계약(S4.4 결박).** ③의 판별자는 "ManagerItem 처분(status open→resolved/dismissed)과 Request 진행이 한 transaction에서 원자로 일어난다"는 §9 Q5 전이 원자성에 기댄다 — 그래야 committed 상태에 "ManagerItem open인데 Request 진행" 같은 창이 없다. S4.4가 FromDeadlock Item을 처분할 때 Item 처분과 Request 전이를 반드시 같은 transaction에 써야 하며, 이는 새 제약이 아니라 기존 aggregate 전이 원자성의 적용이다. §9 Q3③의 "replay 멱등 창은 downstream 미진행에 결박"과 같은 결로, S4.3d는 처분 뒤 상태를 손상으로 오탐하지 않는다. S4.4가 durable Case를 escalated에서 다른 status로 바꾸지 *않는다*(Case escalation은 terminal·①의 양방향 결박이 영구)는 것도 이 게이트의 전제다 — S4.4 처분은 ManagerItem·Request를 진행시키되 Case status는 escalated로 남긴다.

- **불변식 보존.** **전이 ≠ 기록** — S4.3d는 read-only 증명이라 전이도 기록도 쓰지 않는다(repair 0). **미아 없음** — ①②가 escalated Case → receipt → 반드시 존재하는 FromDeadlock ManagerItem(유일 Manager 또는 root 큐)까지 durable 도달을 검증한다. **Authority 중앙** — 검증만 하고 어떤 권한·상태도 선언하지 않는다. **노출 불변식** — report는 typed ref·enum·정수만 담고 raw claim·Manager/root User·원문 0.

### 11. S4.4 durable Manager 처분의 FromDeadlock 소비 계약 — read-only escalation receipt 결박·Case 불변 (2026-07-22 보강)

§9(c.3)가 FromDeadlock ManagerItem을 `status="open"`으로 *생성*까지만 하고 처분(Assign/Dismiss)을 S4.4로 이월했으므로, S4.4d는 그 open Item을 durable 처분으로 소비한다. 새 되돌리기 어려운 도메인 결정은 없다 — escalation의 사람 승인은 c.3가 이미 취득·소비했고 Case escalation의 terminal 성격은 §10이 정했다. S4.4d는 그 계약들 위에서 "처분이 무엇을 evidence로 삼는가(③)"와 "Case를 건드리지 않음(D)"만 확정하므로 자기 ADR 없이 이 보강으로 둔다. 처분의 중앙 role·resource 계약은 ADR 0050 §12다. tdd 소유 모듈은 신규 `sqlite_durable_manager_disposition_uow.py`(FromUnowned b/c와 공유·source별 분기).

- **③ evidence proof = committed escalation receipt graph의 read-only 결박(새 HITL 0).** InMemory deadlock 처분(`p17_deadlock_manager_disposition`)은 원 Conflict claim과 Manager claim을 full handle로 묶은 mediation proof가 저장된 뒤에만 처분했다. durable 판의 동형은 **FromDeadlock Item이 committed escalation receipt(c.2 graph)에 durable하게 결박됨을 read-only로 증명**하는 것이다. escalation의 사람 승인(HITL)은 c.3가 이미 취득·소비했으므로 처분은 승인을 재취득하지 않는다 — 처분은 Manager의 routine `manager.act`(1인칭·ADR 0050 §12)다. UoW는 c.2 escalation receipt를 `(org_id, request_id)`로 조회해 (a) 존재, (b) `source:sha256(conflict_id)`가 Item의 `source_ref`와 일치, (c) `awaiting_revision`이 Item과 일치를 확인한다. 이 결박은 승인된 escalation 없이 위조된 orphan deadlock Item이 처분되는 것을 막는다(§10 reconciliation의 receipt↔ManagerItem 결박을 명령 시점에 재확인). 새 HITL 승인 증거는 만들지 않는다.
  - **기각한 대안 (b) 처분마다 새 HITL 승인 증거:** escalation 승인(operator·c.3)과 처분(manager·routine)을 혼동하고, ADR 0050 §12가 `manager.act`를 사람 승인 2층 없는 1인칭 처분으로 정한 것과 어긋난다. 처분은 escalation 승인의 재취득이 아니라 그 committed 결과의 소비다. InMemory의 mediation proof도 새 승인이 아니라 두 claim의 결박 재확인이었다.

- **D Case 불변 — 처분은 ManagerItem·Request만 전이, Case는 escalated terminal 고정.** InMemory deadlock 처분은 `_resolve_case`(Assign)·`_decline_case`(Dismiss)로 ConflictCase를 resolved/declined로 바꿨다. durable 판은 이를 **복사하지 않는다** — §10이 `receipt(org,conflict_id) 존재 ⟺ Case escalated`를 terminal·영구 불변식으로 못박았고, S4.3d reconciliation의 ①(양방향 결박)이 escalated Case를 다른 status로 바꾸는 것을 손상으로 잡는다. 따라서 S4.4d FromDeadlock 처분은 **`durable_linked_conflict_cases`를 읽지도 쓰지도 않고**(evidence 결박은 c.2 receipt로), ManagerItem(open→resolved/dismissed)과 Request(AwaitingManager(public_kind="contested")→ReadyToDispatch/DeclinedRequest)만 전이한다. §10 말미의 "S4.4가 durable Case를 escalated에서 다른 status로 바꾸지 *않는다*"가 이 결정의 downstream 결박이다. S4.4d 완료 판정은 처분 후 S4.3d reconciliation green(Case escalated 불변·ManagerItem resolved/dismissed·Request 진행 tolerant)을 포함한다.

- **불변식 보존.** **미아 없음** — Assign은 같은 Request를 실행 가능한 `ReadyToDispatch`로, Dismiss는 중립 `DeclinedRequest(manager_declined)`로 닫아 escalated deadlock을 종착시킨다. **Authority 중앙** — 처분 role(manager·ADR 0050 §12)·evidence 결박(c.2 receipt)은 중앙에서만 온다. **전이 ≠ 기록** — ManagerItem·Request 전이(도메인)와 S4.1 command receipt/audit/outbox(기록)를 한 transaction에 함께 쓰되 개념은 구분한다. **노출 불변식** — Dismiss는 고정 `manager_declined` reason_code·중립 메시지만 투영하고 rationale은 durable하게 지속하지 않는다.

### 12. S4.6 linked command-receipt reconciliation gate — committed 처분/발급 command receipt ⟺ target aggregate·Request 상태의 read-only 증명 (2026-07-24 보강)

S4.4(§11·ADR 0050 §12)가 `manager.assign_owner`/`manager.dismiss`를, S4.5(ADR 0042 §5·§8)가 `work_ticket.create`를 S4.1 `durable_linked_command_receipts`에 쓰는 유일한 writer로 닫았으므로, S4.6은 그 커밋된 command receipt가 target aggregate·Request 상태와 실제로 정합함을 **명령·transaction·repair 없이 read-only로 증명하는 게이트**를 연다. §10(escalation reconciliation)이 c.2 escalation receipt graph(별 component)를 대상으로 한 것과 대칭이며, **§10의 "ManagerItem.status 판별자로 정상 downstream 진행을 손상과 구분한다"는 판을 S4.1 command receipt 전반으로 일반화**한 것이다. 새 되돌리기 어려운 도메인 결정은 없다 — 검증할 결박(manager.assign_owner⟺Item resolved+Request ReadyToDispatch|이후 / manager.dismiss⟺Item dismissed+Declined / work_ticket.create⟺WorkTicket pending|이후+AwaitingAnswer|이후)은 §11·ADR 0050 §12·ADR 0042 §5·§8이 이미 선언했고, S4.6은 그것을 읽어 대조할 뿐이다. 따라서 자기 ADR 없이 이 보강으로 둔다(§10 S4.3d·S4.2 reconciliation·S3.2d one-shot reconciliation이 자기 ADR 없이 처리된 선례와 같다). tdd 소유 모듈은 신규 `sqlite_durable_linked_reconciliation.py`(S4.1의 read-only sibling — §10 판과 같이 write 표면 0·별 모듈 신설).

- **Q1 — 증분 경계(S4.1 `_validate_rows`·§10 S4.3d·각 UoW replay와 겹치지 않는 S4.6 고유 검증).** S4.1 `_validate_rows`는 command receipt의 *행 문법*(typed ref·SHA·enum·정수·timestamp)·*intent mirror*(audit/outbox 1:1)·*org/request lineage*까지만 보고 **receipt.action ⟺ 대상 aggregate status ⟺ Request.state는 보지 않는다**. §10 S4.3d는 *별 component*(`durable_conflict_escalation_receipts`)와 Case escalated·FromDeadlock Item *생성*만 보며 `durable_linked_command_receipts`·Item *처분* 결박·WorkTicket을 보지 않는다. 각 UoW replay(`_stored_result`)는 *한 명령·gated·write transaction 안·단일 request*만 본다. S4.6의 고유 증분은 **org-scope 전수 sweep으로, 명령 없이, downstream 진행을 허용하며** command receipt ⟺ aggregate·Request 상태 정합을 증명하는 것이다.
  - **① forward(receipt-anchored) — action별 대상 결박.** 모든 committed `durable_linked_command_receipts`(org) row를 action으로 분기한다.
    - **`manager.assign_owner`** → target_ref(==`manager_item_id` 그대로·재hash 없음)로 ManagerItem 존재·org/request_id 일치·`status == resolved`(직접 target·**terminal 영구**). Request는 `revision >= receipt.expected_request_revision + 1`(전이 floor) + **판별자=Request.state 유형**: 아직 `ReadyToDispatch`이고 그 `trigger_key == receipt_id`(이 receipt가 현재 resting 원인)이면 정확한 resting shape(`attempt == 1`·route 존재·`trigger_key == receipt_id`·handling(kind=`system`, ref==`receipt_id`))여야 하고, 그 너머로 진행했으면(다른 state·다른 trigger — S4.5 소비로 `AwaitingAnswer` 등) revision floor·org만 요구한다.
    - **`manager.dismiss`** → ManagerItem `status == dismissed`(terminal 영구). Request는 `DeclinedRequest(reason_code="manager_declined")`로 정확히 결박(**terminal이라 downstream 진행 없음** — revision == expected+1). tolerance 불요.
    - **`work_ticket.create`** → target_ref(==`ticket_id` 그대로)로 WorkTicket 1:1 join·org/request_id 일치·`awaiting_revision == receipt.expected_request_revision`·`status ∈ {pending, completed, escalated}`(**forward-tolerant** — 오늘은 pending만 쓰이나 S5 lease/delivery가 completed/escalated로 진행). Request는 `revision >= expected+1` + **판별자=state 유형**: 아직 `AwaitingAnswer`이고 `ticket_id == target_ref`이면 정확한 shape(`attempt == ticket.attempt`·route_sha256 == ticket.route_sha256·handling(kind=`runtime_ticket`, ref==`ticket_id`))여야 하고, 진행했으면(Answered/Failed 등) revision floor·org만 요구한다. attempt는 receipt에 없고 ticket·Request resting shape에서만 대조한다.
    - **그 밖의 action/target**(`conflict.agree`·`conflict.escalate`·`manager.reroute` 또는 target_kind가 `manager_item`/`work_ticket`가 아님) — 현재 이 테이블에 writer가 없다. S4.6이 결박할 수 없는 receipt는 fail-closed로 `unbindable_command_receipt` violation(미탐 0 — 새 writer가 생기면 S4.6도 함께 확장). 
  - **② backward(aggregate-anchored) — 처분/발급의 receipt 필수(receipt 없는 partial state 검출).** (a) 모든 `status ∈ {resolved, dismissed}` ManagerItem(org) → status에 맞는 command receipt(resolved⟺`manager.assign_owner`·dismissed⟺`manager.dismiss`)가 target_ref==manager_item_id·org/request 일치로 정확히 하나 존재해야 한다. 없으면 `disposed_item_without_receipt`(스펙의 "dismissed item인데 receipt 0"이 이 arm). **status=open Item은 command receipt를 요구하지 않는다** — 처분 전이며, 생성은 escalation(§10 S4.3d)·FromUnowned ingress 소관. (b) 모든 WorkTicket(org·status 무관) → `work_ticket.create` receipt가 target_ref==ticket_id·org/request/attempt 일치로 존재해야 한다. 없으면 `work_ticket_without_receipt`(WorkTicket은 항상 receipt와 같은 transaction에서만 생기므로 receipt 없는 ticket = partial/위조).
  - **③ downstream tolerance 판별자 — §10 ManagerItem.status 판의 일반화.** §10은 ManagerItem.status(open↔resolved/dismissed)를 "S4.4 처분 발생 여부"의 authoritative 신호로 삼아 Request 진행을 관용했다. S4.6은 이를 **2층**으로 일반화한다 — **(직접 target aggregate)** receipt의 target_ref가 가리키는 aggregate status는 exact이고 monotone이다(ManagerItem resolved/dismissed는 영구·WorkTicket pending→completed→escalated는 forward-only), 그래서 처분/발급 자체는 정확히 검증한다. **(downstream Request)** Request는 tolerant이며, 판별자는 **"Request가 아직 이 receipt의 resting 지점에 있는가 = `revision == receipt.expected_request_revision + 1`"**(resting-revision 판별)이다 — 그렇다면 정확한 resting shape(kind·identity[trigger_key/ticket_id]·attempt·route/handling 전부)을 요구하고, `revision > expected + 1`(진행)이면 revision floor + org만 요구한다. **(정밀화 2026-07-24·독립 리뷰 발견)**: 원안의 "kind AND identity가 이 receipt를 가리키는가" 판별은 두 방향 모두 부정확했다 — (i) kind-only는 ticket 다중 attempt(`UNIQUE(request_id, attempt)`·상태기계의 `ReadyToDispatch(attempt+1)` 재시도 경로가 이미 모델링)에서 receipt-1 sweep이 ticket-2를 가리키는 정상 Request를 손상으로 **오탐**하고, (ii) identity-AND는 revision이 resting인 채 identity만 변조된 drift-at-rest를 관용해 **미탐**한다. resting-revision 판별은 둘 다 해소한다: 진행한 Request(revision 초과)는 관용(오탐 0), resting인데 어떤 필드든 어긋나면 손상(미탐 0). 이 판별은 §9 Q5·§10의 "aggregate 처분·Request 전이·receipt가 한 transaction 원자"라는 전제에 기댄다 — 그래서 "receipt 있는데 aggregate/Request 미진행" 같은 committed 창이 없어 오탐 0, "처분됐는데 receipt 0"·"receipt 있는데 aggregate 결손" 같은 실제 partial state는 미탐 0으로 잡는다.

- **Q2 — 소관 경계(S4.2b concurrence·c.2 escalation receipt 재검증 없음).** S4.6은 `durable_linked_command_receipts`(S4.4/S4.5 writer)만 anchor로 삼는다. **c.2 escalation receipt**(`durable_conflict_escalation_receipts`·별 component)와 그 Case escalated·FromDeadlock Item *생성*은 §10 S4.3d가 소유하므로 S4.6은 escalation graph를 읽지도·재검증하지도 않는다. 둘의 유일한 접점은 FromDeadlock Item의 *처분*이다 — S4.4가 그 open Item을 `manager.assign_owner`/`dismiss`로 처분하면 command receipt가 남고, 그 처분⟺command receipt 결박이 S4.6 몫이다(S4.3d는 그 처분을 status-tolerant로 관용만 하고 receipt 뒷받침은 증명하지 않는다 — 상보·무중복). **S4.2b concurrence receipt**(`durable_direct_conflict_receipts`·자기 component·자기 replay 검증)는 S4.6이 읽지 않는다. S4.6은 **ConflictCase를 아예 대상으로 하지 않는다** — 실 writer의 command receipt target_kind는 `manager_item`·`work_ticket`뿐이며 `conflict_case`를 쓰지 않고, Case의 open/resolved/escalated는 각각 ingress·S4.2b·S4.3c/d 소관이다.

- **Q3 — 모듈·report·fail-closed·org-scope.** 별 모듈 신설(`sqlite_durable_linked_reconciliation.py`) — cross-aggregate reader가 command_receipts × manager_items × work_tickets × question_requests state를 가로지르므로 S4.1 schema 모듈에 넣으면 S4.1이 Request·aggregate *state*에 의존하게 돼 소유권이 샌다(§10 Q2·S3.2d 판). report는 `DurableLinkedReconciliationReport(capable, detail, linked_aggregates_manifest_present, violations)`, violation은 typed(`anchor_ref`=stored typed ref[receipt_id/manager_item_id/ticket_id/request_id]·`kind`∈{`manager_disposition_receipt_mismatch`, `work_ticket_receipt_mismatch`, `request_state_inconsistent`, `unbindable_command_receipt`, `disposed_item_without_receipt`, `work_ticket_without_receipt`, `linked_capability_uncertain`}·`detail`). `capable == True ⟺ org-scope violation 0`(fail-closed 이진 게이트). **capability-우선:** S4.1 linked aggregates·Completion을 org_id와 함께 먼저 read-only validate(**두 validator만** — S4.6은 escalation graph를 읽지 않아 c.2 validator 불요) — 하나라도 실패하면 어떤 row도 신뢰 못하므로 즉시 `capable=False`·`linked_capability_uncertain` 단일 violation·row 열거 0(전역 corrupt catalog는 모두에게 닫힘). 서면 org-필터 row를 전수 열거하고 모든 violation을 담는다(첫 위반에서 멈추지 않음). **org-scope:** row 열거·cross-aggregate 조회는 org-필터, catalog/manifest/FK는 전역(S4.1 validator에 위임). repair·write·전이 0·open connection handle 노출 0(마지막 게이트).

- **Q4 — one-shot 경계.** 입력은 `db_path`·`org_id: str | None = None`뿐(principal·authorizer·factory 불요). cursor·scheduler·lease·wake·delivery 0. `mode=ro` connection·`PRAGMA foreign_keys=ON` 뒤 **하나의 deferred read transaction(`BEGIN`…`COMMIT`)으로 전 sweep을 한 snapshot에 묶어**(receipt·ManagerItem·WorkTicket·Request가 서로 다른 시점을 보지 않게) 열거·검증·close한다. typed Request state는 `_select_question_request_no_commit`으로, 나머지는 raw SQL로 읽는다. **typed ref helper 불요** — S4.6은 `target_ref == manager_item_id`·`target_ref == ticket_id`처럼 저장된 typed ref끼리 직접 join한다(S4.4/S4.5 writer의 "target_ref=원본 ID 그대로·재hash 금지" 규약이 이 join을 단순 등호로 만든다). c.3/S4.1 값 객체를 import하지 않는다.

- **불변식 보존.** **전이 ≠ 기록** — read-only 증명이라 전이도 기록도 쓰지 않는다(repair 0). **미아 없음** — ②가 처분된 Item·발급된 WorkTicket이 반드시 command receipt에 durable 결박됨을, ①이 그 receipt가 실행 가능한 Request 진행(`ReadyToDispatch`/`AwaitingAnswer`)이나 중립 종착(`DeclinedRequest`)으로 이어졌음을 증명한다. **Authority 중앙** — 검증만 하고 어떤 권한·상태도 선언하지 않는다. **노출 불변식** — report는 typed ref·enum·정수만 담고 raw claim·User·원문 0.

## Consequences

- **불변식 영향 없음:**
  - **Authority 중앙** — 승인 증거는 사람 통제(HITL)이지 카드 자기보고 authority가 아니다. 명령 role(operator)·Manager/root 선택(c.0 graph)은 중앙 Registry/정책에서 온다. 어떤 카드도 escalation authority를 자기보고하지 않는다.
  - **전이 ≠ 기록** — c.1은 shape/계약만이며 전이도 write도 없다. Case escalated·ManagerItem·Request 전이(도메인)와 receipt/audit/outbox(기록)는 각각 S4.3c.3·S4.3c.2다.
  - **미아 없음** — escalation은 deadlock/drift된 Conflict를 사람(Manager)에게 durable하게 닿게 하는 경로다. 증거 계약은 미아를 만들지 않는다.
  - **노출 불변식** — 증거는 secret-free다. digest·fingerprint·evidence_id만 담고 raw claim·Manager/root User ID·원문은 담지 않는다(S4.3b·c.0 결과와 같은 결).
- **DB write 0·전이 0·실 store 0.** S4.3c.1은 read-only 계약·포트·순수 검증만 연다.
- **후속 결정 대기:** 파일럿 조직 승인자 구체 신원, 실 HITL 승인 UI, 만료 TTL·취소·재승인의 durable schema(c.2)·평가 transaction(c.3), durable policy epoch·다중 인스턴스 일관성.

## S4.3c.1 RED 인수조건 (shape)

- `ConflictEscalationApprovalEvidence`는 frozen이고, `command_digest`·`resource_fingerprint`·`escalation_cause_digest`·`graph_selection_digest`가 각 64 hex가 아니거나 `evidence_id` blank거나 `action`/`status` 리터럴이 아니면 생성 실패다.
- `canonical_escalate_command_digest`는 같은 action·resource·command에 같은 digest를, 다르면 다른 digest를 낸다. `escalation_cause_digest`는 같은 S4.3b sealed evidence에 결정론 digest를 낸다.
- `graph_selection_digest`는 S4.3c.0 snapshot의 `graph_digest`와 exact 일치한다.
- 순수 검증: granted 증거의 4결박이 현재 명령·resource·cause·graph와 exact일 때만 유효, 하나라도 어긋나면 무효. 취득은 1회, same-evidence 재확인은 취득 직후·write 직전 2회 모두 통과해야 유효, replay는 승인 재취득 없이 current만 재검증.
- escalation 증거 모듈은 `hitl` toggle 모듈을 import하지 않는다.
